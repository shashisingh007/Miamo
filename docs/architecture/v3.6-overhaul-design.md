# 1. Executive summary

## 1.1 The bet

In one sentence: **make Miamo the only dating app that knows the user *right now*, writes openers in their voice, gives every active user fair visibility, and ships features no competitor has — for India first.**

## 1.2 What's being built

**Eight tightly-scoped systems**, every one default-OFF until ramped:

1. **Real-time intent + mood inference (§A)** — 7-class intent vector + 5-dim mood vector + polarity classifier + depth-of-engagement classifier. 90-second TTL. Pinterest-style "session vibe" applied to dating. New worker `intentInference.ts`; new pure module `algo/v8/intentRightNow.ts`. Behind `ALGO_V8_INTENT_RIGHT_NOW_ENABLED` (default OFF) + `Settings.moodInferenceEnabled` (default OFF — explicit consent gate).

2. **Exposure ledger + weekly Top-10 stable-match (§B)** — Singh-Joachims fairness math (cited from arxiv 1802.07281). New `ExposureLedger` Prisma model mirrors `SpotlightLedger`. Weekly Gale-Shapley pass writes the curated Top-10 — the Hinge math, made visible. Gender-conditional Gini target ≤0.40 (corrects for Indian gender skew). Premium 1.5× multiplier max.

3. **Multi-objective filter ranking (§B)** — replaces within-pool jitter with `relevance × earnedVisibility × fairness × recencyFreshness × intentFitRightNow`. Per-surface weights.

4. **Move v2 composer (§C)** — sender-voice extraction from last 50 outbound messages, receiver-resonance from last 10 successful replies, 80-template code-mix support (en / Hinglish / Tanglish / Banglish), 42 forbidden phrases (16 existing + 26 from market scan). **No LLM.** Predicted-reply ranking; <2% fallback rate target. Plus a **"Your voice fingerprint"** reveal UI — share-to-Instagram-Story enabled.

5. **DTM mood-mask + cold-start improvements (§D)** — `dtmFeedV7` gains optional `topicMask` param (pure module stays pure). Heavy topics (intimacy, conflict, finance) gated by mood < 0.4 OR last-2-sessions windowShopping. `dtmColdStart.ts` thresholds sacred.

6. **Family Brief (§D)** — one-tap parent-shareable bio-data card from `MatrimonialProfile`. PDF / image (1080×1920 for WhatsApp story) / plain text. 7-day TTL token. **Zero competitors ship this.** India-first viral mechanic.

7. **Anti-ghost economy (§D)** — sender deposits 1 Spotlight minute starting a chat; receiver gets it on first reply within 72h; otherwise burns. Premium 50% deposit. Re-uses existing ledger.

8. **Voice-note chat + transparency UI (§D + §E)** — voice-message UI in messaging surface (WhatsApp-transition fix per India market). Plus card-level "**why am I seeing this**" explainer (3-star ingredient breakdown, Article 22 compliant, no competitor exposes their ranker).

Plus **two cross-cutting deliverables**:

9. **v3.5.1 hotfix (§B.1)** — hard-filter passed profiles 30d. Ships BEFORE the overhaul AB. One-line query change. Expected: 20-30% reduction in #1 user complaint.

10. **Patent clearance + DPIA (§E)** — 6-day Week-0 process. Three design-around paths if Match Group has issued/pending patents on session-intent classification.

### A.0 Why this exists (60 seconds)

The single biggest gripe from every dating-app megathread is "it doesn't learn me" (MARKET_SCAN §2, user pain #11; complaint #1 "shows me the same people"). Miamo's existing `rightNow.ts` (V7 phase G) blends four short-horizon signals into a single 0..1 surface-momentum score. That is useful but flat — one number cannot tell the ranker the difference between *"user is on a focused serious-search session at 11pm on a Thursday"* and *"user is hate-scrolling at lunch."* Both surface high `score` today. The downstream effect is wrong candidates in both cases.

Section A replaces the flat scalar with a structured perception triple — **(IntentVector, MoodVector, Polarity)** — plus a unit-signal classifier — **DepthOfEngagement** — that replaces raw impression counts everywhere they currently flow into rankers and fairness math. All four are pure modules. All four are derived in a single worker pass (`intentInference.ts`) on a 30s/5min schedule. All four are persisted only in `FeatureSnapshot.raw` JSONB with a 90s TTL on the volatile fields. No new Prisma tables. No new write-heavy infra. No LLM.

The four artefacts:

| Artefact | Shape | TTL | Storage |
|---|---|---|---|
| `intentVec` | 7-class probability vector (sum=1.0) | 90s | `FeatureSnapshot.raw.intentRightNow` |
| `moodVec` | 5-dim 0..1 vector | 90s | `FeatureSnapshot.raw.moodRightNow` |
| `polarity` | scalar -1..+1 per recent dwell | per-event | computed inline; not stored |
| `depthScore` | scalar 0..1 per impression | per-event | computed inline; written into `EventAggHourly` via rollup |

The compose pattern (`compose(breakdown, weights)`), the universal fatigue penalty (`2 * log1p(impressions48h)`), the stable jitter (`stableJitter(viewerId, candidateId, 5*60_000)`), and the learner ramp (`applyLearnerRamp`) are not reinvented. Section A produces *features*; the rankers in Section B consume them.

### A.1 Right-now intent v8 (extends `rightNow.ts`)

#### A.1.1 The seven intent classes

The founder asked for "what does the user need right now." We translate that into a 7-class probability vector. The classes are mutually-exclusive-at-the-mode but compositional in the vector (a session can be 60% casual-scroll, 30% intentional-browse, 10% reply-mood — that vector tells the ranker something different from "casual = 1.0").

Class names match the §8.1 list in the prompt verbatim. Definitions, signature signals, default confidence below.

| # | Class | Definition | Signature signals (positive evidence) | Default confidence on cold-start |
|---|---|---|---|---|
| 1 | **distraction-browse** | User opened the app to fill 30 seconds. No commitment, fast flicks, low dwell, frequent app-switches. | `card.impression.50.count / minute >= 12`, median `dwellMs < 600`, `nav.route` churn >= 4/min, no `card.bio.expand` in last 90s | 0.10 |
| 2 | **intentional-browse** | User is actively looking but hasn't committed to acting. Medium dwell, occasional bio expand, no swipe spree. | median `dwellMs` in [1500, 6000], `card.bio.expand >= 1` per 90s, swipe rate in [3, 8]/min, `card.photo.swipe` present | 0.15 |
| 3 | **reply-mood** | User came to respond to existing matches, not discover new ones. | `nav.route` to `/messages` within last 120s, `msg.send` rate > `swipe.commit` rate over last 5m, `focus.element` on compose box | 0.10 |
| 4 | **review-existing** | User is revisiting old matches, see-later pile, or saved profiles. | `discover.see_later.view` events present, `nav.route` to `/matches`, repeated `tid` in card impressions, `card.bio.expand` on previously-seen `tid` | 0.05 |
| 5 | **serious-search** | User is in deliberate evaluation mode. Long dwells, multiple bio expands, photo swipes through full carousel, hesitation. | median `dwellMs > 5000`, `card.bio.expand >= 2` per 90s, `filter.hesitation` events recent, `card.photo.swipe.toIndex >= 3` on multiple cards | 0.10 |
| 6 | **casual-scroll** | The neutral default — pleasant browsing without strong commitment either way. | median `dwellMs` in [800, 2500], swipe rate in [4, 10]/min, mixed left/right, low rage | 0.45 |
| 7 | **decision-fatigued** | User is tired, agitated, swiping aimlessly, may regret. Cut-off signal. | `swipe.repeat_pass` events present, `swipe.regret` recent, rage-clicks > 4/5min, dwell variance high, session length > 12m | 0.05 |

Sum of default cold-start confidences = 1.00. // because: a valid probability vector must sum to 1.0; cold-start is the prior the Bayesian update walks away from.

#### A.1.2 Classification logic — pure module

New file: `services/shared/src/algo/v8/intentRightNow.ts`. Pure. No I/O. Deterministic. Tested by the contract test in `registerAlgo({...usesEvents:[...]})`.

```ts
// services/shared/src/algo/v8/intentRightNow.ts
import { clamp01 } from '../math';
import { registerAlgo } from '../registry';

export type IntentClass =
  | 'distractionBrowse'
  | 'intentionalBrowse'
  | 'replyMood'
  | 'reviewExisting'
  | 'seriousSearch'
  | 'casualScroll'
  | 'decisionFatigued';

export type IntentVector = Record<IntentClass, number>; // sum = 1.0

export type RecentEvent = {
  name: string;           // v6/v7/v8 event name
  ts: number;             // epoch ms, client-local
  payload: Record<string, unknown>;
};

export type ViewerFeatures = {
  hourTotalsLocal: number[];        // 24-len, lifetime by-hour activity (existing)
  recentRouteChurn: number;         // nav.route count last 90s (existing rollup)
  recentDwellMsP50: number;         // last 90s
  recentDwellMsVar: number;         // variance, last 90s
  recentSwipesPerMin: number;
  recentMsgSendsPerMin: number;
  recentRageClicks: number;
  recentBioExpands: number;
  recentPhotoSwipes: number;
  recentSeeLaterViews: number;
  recentRegretCount: number;
  recentRepeatPassCount: number;
  recentHesitationCount: number;
  sessionAgeMs: number;
  isOnMessagesRoute: boolean;
  isOnMatchesRoute: boolean;
  sessionsAccrued: number;          // total sessions ever — for cold-start gate
};

// Per-class log-odds contribution weights. Pure heuristic — tunable post-launch.
// because: a 7-way log-linear classifier with hand-picked features is intelligible
// and falsifiable; we keep room for a learned MLP later behind a separate flag.

const COLDSTART_PRIOR: IntentVector = {
  distractionBrowse: 0.10, // because: minority intent, baseline noise
  intentionalBrowse: 0.15,
  replyMood:         0.10,
  reviewExisting:    0.05,
  seriousSearch:     0.10,
  casualScroll:      0.45, // because: this is the population-mode for a healthy app
  decisionFatigued:  0.05,
};

// Feature → class log-odds matrix. Comments cite the signal source.
// Formula numbers map to A.1.3 below.

export function classifyIntentRightNow(
  recent: ReadonlyArray<RecentEvent>,
  features: ViewerFeatures,
): IntentVector {
  // Cold-start gate: first 5 sessions = pure prior; see A.7.
  // because: with < 5 sessions of telemetry every classifier is dominated by noise
  if (features.sessionsAccrued < 5) return { ...COLDSTART_PRIOR };

  const logits: IntentVector = { ...COLDSTART_PRIOR };
  // Convert prior into log-space (we exponentiate + renorm at the end).
  (Object.keys(logits) as IntentClass[]).forEach(k => { logits[k] = Math.log(logits[k]); });

  // ── (1) distraction-browse contribution
  if (features.recentRouteChurn >= 4)         logits.distractionBrowse += 0.8;
  if (features.recentDwellMsP50 < 600)        logits.distractionBrowse += 0.9;
  if (features.recentSwipesPerMin >= 12)      logits.distractionBrowse += 0.6;
  if (features.recentBioExpands === 0)        logits.distractionBrowse += 0.3;

  // ── (2) intentional-browse contribution
  if (features.recentDwellMsP50 >= 1500
   && features.recentDwellMsP50 <= 6000)      logits.intentionalBrowse += 1.0;
  if (features.recentBioExpands >= 1)         logits.intentionalBrowse += 0.7;
  if (features.recentPhotoSwipes >= 1)        logits.intentionalBrowse += 0.5;
  if (features.recentSwipesPerMin >= 3
   && features.recentSwipesPerMin <= 8)       logits.intentionalBrowse += 0.4;

  // ── (3) reply-mood contribution
  if (features.isOnMessagesRoute)             logits.replyMood += 1.4;
  if (features.recentMsgSendsPerMin > features.recentSwipesPerMin) logits.replyMood += 1.0;
  if (features.recentMsgSendsPerMin >= 2)     logits.replyMood += 0.6;

  // ── (4) review-existing contribution
  if (features.isOnMatchesRoute)              logits.reviewExisting += 1.2;
  if (features.recentSeeLaterViews >= 1)      logits.reviewExisting += 1.5;
  // Repeated `tid` impressions detected from `recent` event list
  const tids = new Set<string>();
  let repeatTidHits = 0;
  for (const ev of recent) {
    const tid = (ev.payload as { tid?: string }).tid;
    if (!tid) continue;
    if (tids.has(tid)) repeatTidHits++;
    tids.add(tid);
  }
  if (repeatTidHits >= 2) logits.reviewExisting += 0.8;

  // ── (5) serious-search contribution
  if (features.recentDwellMsP50 > 5000)       logits.seriousSearch += 1.2;
  if (features.recentBioExpands >= 2)         logits.seriousSearch += 1.0;
  if (features.recentPhotoSwipes >= 3)        logits.seriousSearch += 0.7;
  if (features.recentHesitationCount >= 1)    logits.seriousSearch += 0.6;

  // ── (6) casual-scroll contribution — neutral pull on most features
  if (features.recentDwellMsP50 >= 800
   && features.recentDwellMsP50 <= 2500)      logits.casualScroll += 0.6;
  if (features.recentSwipesPerMin >= 4
   && features.recentSwipesPerMin <= 10)      logits.casualScroll += 0.3;

  // ── (7) decision-fatigued contribution
  if (features.recentRepeatPassCount >= 1)    logits.decisionFatigued += 1.3;
  if (features.recentRegretCount >= 1)        logits.decisionFatigued += 1.0;
  if (features.recentRageClicks >= 4)         logits.decisionFatigued += 1.4;
  if (features.recentDwellMsVar > 8_000_000)  logits.decisionFatigued += 0.5; // because: huge dwell-time spread = oscillating between commit and quit
  if (features.sessionAgeMs > 12 * 60_000)    logits.decisionFatigued += 0.4;

  // Softmax to a valid probability vector.
  const maxLogit = Math.max(...Object.values(logits));
  let z = 0;
  const probs: IntentVector = { ...COLDSTART_PRIOR };
  (Object.keys(logits) as IntentClass[]).forEach(k => {
    probs[k] = Math.exp(logits[k] - maxLogit);
    z += probs[k];
  });
  (Object.keys(probs) as IntentClass[]).forEach(k => {
    probs[k] = clamp01(probs[k] / (z || 1));
  });

  return probs;
}

registerAlgo({
  name: 'intentRightNowV8',
  usesEvents: [
    'card.impression.50', 'card.impression.100', 'card.hover',
    'card.bio.expand', 'card.bio.collapse', 'card.photo.swipe',
    'discover.swipe', 'swipe.commit', 'swipe.regret', 'swipe.repeat_pass',
    'nav.route', 'focus.element', 'msg.send',
    'discover.see_later', 'discover.see_later.view',
    'filter.hesitation', 'attention.idle.enter', 'attention.idle.exit',
  ],
  surface: 'foundation',
});
```

#### A.1.3 Formulas (numbered, for reproducibility)

1. **Distraction-browse log-odds:**
   `Δ_dist = 0.8·[churn≥4] + 0.9·[dwellP50<600] + 0.6·[swipes/min≥12] + 0.3·[bioExpands=0]`
2. **Intentional-browse log-odds:**
   `Δ_int = 1.0·[1500≤dwellP50≤6000] + 0.7·[bioExpands≥1] + 0.5·[photoSwipes≥1] + 0.4·[3≤swipes/min≤8]`
3. **Reply-mood log-odds:**
   `Δ_reply = 1.4·[onMessagesRoute] + 1.0·[msgRate>swipeRate] + 0.6·[msgSends/min≥2]`
4. **Review-existing log-odds:**
   `Δ_review = 1.2·[onMatchesRoute] + 1.5·[seeLaterViews≥1] + 0.8·[repeatTidHits≥2]`
5. **Serious-search log-odds:**
   `Δ_serious = 1.2·[dwellP50>5000] + 1.0·[bioExpands≥2] + 0.7·[photoSwipes≥3] + 0.6·[hesitations≥1]`
6. **Casual-scroll log-odds:**
   `Δ_casual = 0.6·[800≤dwellP50≤2500] + 0.3·[4≤swipes/min≤10]`
7. **Decision-fatigued log-odds:**
   `Δ_fat = 1.3·[repeatPass≥1] + 1.0·[regret≥1] + 1.4·[rageClicks≥4] + 0.5·[dwellVar>8M] + 0.4·[sessionAge>12min]`
8. **Posterior:**
   `P(class | features) = softmax(log(prior_class) + Δ_class)` clamped to [0,1] and renormalised so `Σ_class P = 1.0`.

All coefficients (`0.8`, `0.9`, `0.6`, `0.3`, `1.0`, `0.7`, `0.5`, `0.4`, `1.4`, `1.5`, `1.2`, `1.3`) are exposed as `INTENT_V8_COEFF_*` env vars with the listed defaults. // because: market-scan §6 (privacy) and the prompt §6.3 require any feeling-arbitrary constant to be either cited or env-tunable; we have no published Miamo telemetry yet, so env-tunable is the honest choice.

### A.2 Mood vector v8 — 5 dimensions

The five dimensions match the prompt §8.1 list verbatim: **rage / calm / curious / receptive / fatigued**. Each is independently in [0,1] — it is *not* a probability vector. A user can be simultaneously calm-and-curious or rage-and-fatigued. The vector lives in `FeatureSnapshot.raw.moodRightNow` with the same 90s TTL.

#### A.2.1 Per-dimension computation

```ts
// services/shared/src/algo/v8/moodRightNow.ts
export type MoodVector = {
  rage: number;       // 0..1
  calm: number;       // 0..1
  curious: number;    // 0..1
  receptive: number;  // 0..1
  fatigued: number;   // 0..1
};

export type MoodInput = {
  rageClicksLast5m: number;       // existing — from FeatureSnapshot.rageClickRate × 5
  deadClicksLast5m: number;       // existing
  dwellMsP50Last90s: number;
  dwellMsVarLast90s: number;
  scrollVelocityPxPerSecP50: number;
  hourLocal: number;              // 0..23
  hourTotalsLocal: number[];      // 24-len, lifetime
  sessionAgeMs: number;
  msgSendsLast30m: number;
  msgRepliesReceivedLast30m: number;
  recentBioExpands: number;
  recentRegretCount: number;
};
```

The five dimensions are computed by separate pure functions. // because: independent dimensions = independent bugs; a single matrix would couple them.

Formulas:

9. **rage** = `clamp01( rageClicks/4 + deadClicks/8 + (dwellVar > 8M ? 0.2 : 0) + (regret > 0 ? 0.15 : 0) )`
   // because: rage-clicks are the textbook signal (existing `rageClickRate` is in FeatureSnapshot); we cap each contribution so a single rage-spike doesn't peg the score.

10. **calm** = `clamp01( 1 - rage - (sessionAge > 20min ? 0.3 : 0) )`
    // because: calm is the complement of rage, dampened by session fatigue; we keep it explicit so downstream can use it as a positive weight without inverting rage on every call.

11. **curious** = `clamp01( bioExpands/3 + 0.2·[1000≤dwellP50≤8000] + 0.3·[photoSwipes≥2] )`
    // because: bio-expand is the strongest in-session evidence of "I want to know more"; the dwell band excludes both flick-scrolling and zone-out.

12. **receptive** = `clamp01( msgRepliesReceivedLast30m/4 + msgSendsLast30m/6 + hourBias·0.3 )` where `hourBias = hourTotalsLocal[hour] / max(hourTotalsLocal)`
    // because: receptivity is roughly "is this their usual active hour AND are they in conversation mode"; a user who never logs in at 3am should not be flagged receptive at 3am even if they're momentarily on the messages tab.

13. **fatigued** = `clamp01( (sessionAge/(30*60_000)) + repeatPass/3 + regret/3 + (deadClicks > 4 ? 0.2 : 0) )`
    // because: same machinery as `decision-fatigued` intent class, but as a mood it can co-exist with intent classes other than fatigued (e.g. user is in `serious-search` intent AND `fatigued` mood — a real failure case worth ranking against).

#### A.2.2 Time-of-day prior

`hourBias` is reused from existing `rightNow.ts` (see file, lines 51-59). We do not introduce a new chronotype concept — the existing `FeatureSnapshot.chronotype` row already encodes hour-of-day prior. // because: prompt §6.1 — adds fewer features; matches existing pattern.

#### A.2.3 Recent app history

"Recent app history" in the prompt corresponds to the 30-minute window over `EventAggHourly` (already maintained by the rollup worker). We do not add a new aggregation; we read the existing one and slice by hour.

```ts
};
```

Same TTL contract as intent. Same fallback: stale → neutral `{rage:0, calm:0.5, curious:0.5, receptive:0.5, fatigued:0}`. // because: a user with no data is presumed calm-curious-receptive, not rage-fatigued; the prior is deliberately optimistic to avoid the "mood-tax" anti-pattern (prompt §13).

### A.3 Polarity classifier

The hardest signal in the founder's brief: a long dwell on a profile can mean **"I love this person, let me read every word"** (positive interest → ranker should boost similar candidates) or **"what a disaster, look at this clown"** (hate-scroll → ranker should suppress similar candidates). Today, Miamo treats both as the same impression. That is the root cause of complaint #1 "shows me the same people."

The polarity classifier produces a scalar in `[-1, +1]` for every dwell event of `>= 1500ms`. Short dwells are not classified — there is no time for the signal to disambiguate.

#### A.3.1 Inputs

```ts
// services/shared/src/algo/v8/polarity.ts
export type PolarityInput = {
  dwellMs: number;                       // the dwell event itself
  bioExpanded: boolean;
  photoSwipesDuringDwell: number;
  returnVisitWithin5min: boolean;        // did user come back to this tid?
  finalAction: 'like' | 'pass' | 'super_like' | 'see_later' | 'none';
  msSinceDwellEnd: number;               // for in-flight scoring before action lands
};
```

#### A.3.2 Formula

14. **polarity** =
    `0.40·signOf(finalAction)`
    `+ 0.20·[bioExpanded]`
    `+ 0.15·tanh(photoSwipes / 2)`
    `+ 0.15·[returnVisitWithin5min]`
    `+ 0.10·(dwellMs - 3000) / 6000`
    clamped to `[-1, +1]`.

Where:
- `signOf(like) = +1`, `signOf(super_like) = +1`, `signOf(see_later) = +0.3`, `signOf(pass) = -1`, `signOf(none) = 0`. // because: see-later is positive-but-deferred; we differentiate it from like so we don't over-weight see-later as full commitment.
- The dwell tail term centred at 3s reflects: short-but-positive (1.5s + like) is +0.5-ish; long-and-positive (10s + like + bio-expand) saturates near +1; long-and-pass is the hate-scroll signature and falls below -0.5.

#### A.3.3 Why those weights

- `0.40` on final action: it is the strongest single piece of evidence. Direct user testimony, not inference.
- `0.20` on bio expand: explicit deeper-look behaviour; rarely accidental.
- `0.15` each on photo-swipe and return-visit: corroborating positive signals that don't depend on action.
- `0.10` on dwell tail: contributes texture but doesn't dominate — long dwell alone is ambiguous (the exact problem we are solving).

Sum of positive weights = 1.00. // because: under maximum-positive evidence (all booleans true, action=like, dwell=10s, swipes=4), polarity = +1.00 exactly; symmetric on the negative side via the `pass` action.

All coefficients are env-tunable as `POLARITY_V8_W_*`. // because: same reasoning as A.1.3 — calibrate after 30 days of telemetry.

#### A.3.4 Use site

Polarity is computed **inline at event-ingest time** (when the dwell event is rolled up into `EventAggHourly` and joined with the subsequent action event from the same `tid`). It is *not* stored per-event — only its rolling 30-minute mean per (`uidHash`, `targetSurface`) is persisted into a new field `FeatureSnapshot.raw.polarityRolling30m: number` (single scalar). // because: per-event storage would 100x the row count of impressions; the rolling mean is what every downstream ranker needs anyway.

The rolling mean is updated by the existing rollup loop, not by `intentInference.ts`.

### A.4 Depth-of-engagement classifier

The unit signal of *quality* that replaces raw impression counts across:
- Fairness Gini (KPI 11.8) — currently weighted by impressions; will be weighted by depth-impressions.
- Exposure ledger credit accrual (Section C; out of scope here).
- Fatigue penalty input — `impressions48h` becomes `depthImpressions48h = Σ depth_i`.

The classifier outputs `depth ∈ [0, 1]` for every `card.impression.50` event. The accidental-click filter zeros out the trivial-tap case so phantom impressions (target <5% per KPI 11.10) stop polluting downstream math.

#### A.4.1 Heuristic-first formula

```ts
// services/shared/src/algo/v8/depthOfEngagement.ts
export type DepthInput = {
  dwellMs: number;
  scrollDepthPct: number;       // 0..100, % of card content scrolled
  photoSwipeCount: number;
  bioExpanded: boolean;
  returnCount: number;          // times the same tid was re-impressed within 24h
  undoFlag: boolean;            // user immediately undid a swipe
  photoZoom: boolean;
  screenshot: boolean;
};

export function depthOfEngagement(input: DepthInput): number {
  // ── Accidental-click filter (A.4.2)
  if (input.dwellMs < 500
   && input.scrollDepthPct < 5
   && input.photoSwipeCount === 0
   && !input.bioExpanded) {
    return 0;
  }

  // Named coefficients — see A.4.3.
  const cDwell   = 0.30;  // because: dwell is the universal proxy; not overweighted because it's gameable by leaving the app open
  const cScroll  = 0.20;  // because: scroll-depth is harder to fake than dwell — requires deliberate motion
  const cBio     = 0.20;  // because: bio expand is a strong explicit signal
  const cPhoto   = 0.10;  // because: photo swipe is positive but easy (one finger flick)
  const cReturn  = 0.10;  // because: return visit indicates lingering interest; capped
  const cZoom    = 0.05;  // because: photo zoom is rare and high-signal
  const cScreen  = 0.05;  // because: screenshot is very rare; strong intent

  const dwellTerm  = Math.min(1, input.dwellMs / 8000);              // saturates at 8s
  const scrollTerm = Math.min(1, input.scrollDepthPct / 80);          // saturates at 80%
  const bioTerm    = input.bioExpanded ? 1 : 0;
  const photoTerm  = Math.tanh(input.photoSwipeCount / 2);            // sat smoothly
  const returnTerm = Math.tanh(input.returnCount / 2);
  const zoomTerm   = input.photoZoom ? 1 : 0;
  const screenTerm = input.screenshot ? 1 : 0;

  let depth =
      cDwell   * dwellTerm
    + cScroll  * scrollTerm
    + cBio     * bioTerm
    + cPhoto   * photoTerm
    + cReturn  * returnTerm
    + cZoom    * zoomTerm
    + cScreen  * screenTerm;

  // Undo penalty — strong signal of accidental swipe even with non-zero depth.
  if (input.undoFlag) depth *= 0.4; // because: an undo signals the user didn't mean it; we don't zero because they did see the card

  return Math.max(0, Math.min(1, depth));
}
```

#### A.4.2 Accidental-click filter

Hard rule, named in prompt §10.2: `dwellMs < 500 && scrollDepthPct < 5 && photoSwipeCount === 0 && !bioExpanded ⇒ depth = 0`. The `not a signal` clause means: the rollup writes the impression to `EventAggHourly` (for ops monitoring) but flags it `phantom=true` so the fairness math, exposure ledger, and fatigue penalty all skip it. Phantom-impression rate is KPI 11.10 (target <5%).

#### A.4.3 Coefficient sum

`0.30 + 0.20 + 0.20 + 0.10 + 0.10 + 0.05 + 0.05 = 1.00`. // because: max evidence on every dimension saturates depth at 1.0 exactly.

All seven coefficients exposed as `DEPTH_V8_W_*` env vars with the listed defaults.

#### A.4.4 EWMA tuning plan over 30d telemetry

For the first 30 days post-launch (ramp 0, dark), we run the classifier in *observe-only* mode: depth is computed and logged via `engagement.depth_scored` (event schema in A.5), but no downstream system reads it. After 30 days we have:

1. Joint distribution of `(depth, action)` per user — known good (followed by `msg.send` within 48h) vs known bad (followed by `discover.unmatch` within 7d).
2. Per-coefficient marginal effect on the AUC of `depth → quality-action` prediction.

Tuning runs as:

15. For each coefficient `c_i`, EWMA the daily best-fit value with half-life **14 days** (mirrors existing learner half-life). `c_i^(t+1) = 0.5^(1/14) · c_i^(t) + (1 - 0.5^(1/14)) · best_c_i^(t)`.
16. Hard constraint: `Σ c_i = 1.0` enforced by renormalisation after each step.
17. Tuning loop runs in the existing `learnerLoop.ts` worker, gated by `DEPTH_V8_TUNING_ENABLED` (default 0 until after 30-day observation window).

#### A.4.5 Target accuracy ≥85% on labelled holdout

We will build the labelled holdout from existing data:
- **Positive label (`depth_real=1`):** impressions where the user sent a message within 48h OR completed a DTM topic with the same person OR maintained the match unblocked for ≥14 days.
- **Negative label (`depth_real=0`):** impressions where the user undid the swipe within 60s OR the impression matches the accidental-click filter (A.4.2) OR no further interaction occurred within 30 days.

Holdout split: stratify by user, 80/20, ≥5,000 impressions per class minimum before claiming 85% accuracy. Accuracy is measured as binary classification with threshold `depth >= 0.5 ⇒ predicted=1`.

Tracking: the holdout build is implemented as a one-shot script `scripts/spotlight-depth-labels.py` (mirrors `scripts/spotlight-50-walk.py`), runs after the 30-day observation window, output `scripts/qa-runs/phase-14-depth-accuracy.report.json` with the AUC, accuracy, confusion matrix, and per-coefficient sensitivity.

If accuracy < 85% after EWMA tuning: kick to checkpoint, do not ramp to production weight.

### A.6 New worker job `intentInference.ts`

New file: `services/tracking-worker/src/intentInference.ts`. Default OFF.

#### A.6.1 Schedule

```
for each uidHash in active-user index:
  lastActivityMs = max(EventAggHourly.lastTs WHERE uidHash = X, last 24h)
  ageMs = now - lastActivityMs
  if ageMs < 2 * 60_000:        runEvery = 30 * 1000    // 30s
  elif ageMs < 60 * 60_000:     runEvery = 5 * 60_000   // 5min
  else:                          runEvery = ∞            // sleep — re-queue on next activity event
```

The "active-user index" is a Redis sorted-set `intent:active` keyed by `uidHash`, score = `lastActivityMs`. Updated by `ingest` on every accepted event. The worker pops the head, checks `runEvery`, processes, re-inserts with `lastActivityMs + runEvery`. // because: this matches the existing `learnerLoop.ts` Redis-driven scheduling pattern (knowledge-base §10); no new infra.

A bound `MAX_USERS_PER_TICK = 200` // because: 200 × ~3ms median compute = ~600ms per tick, well under the 30s budget; tunable via `INTENT_INFERENCE_BATCH_SIZE` env.

#### A.6.2 Inputs

For each `uidHash`:
1. `EventAggHourly` last 1h — for `recentDwellMsP50`, `recentSwipesPerMin`, route churn, etc.
2. `UserActivity` last 30min (via the rolled-up tail kept by `rollup.ts`) — for `msgSends`, `seeLaterViews`, `regret`, `repeatPass`.
3. `FeatureSnapshot` — for `chronotype`, `hourTotalsLocal`, `sessionsAccrued`.
4. Settings row for `moodInferenceEnabled` consent flag (A.9).

Reads go through `PrismaSignalReader` (existing) plus a new method `readActivityWindow(uidHash, sinceMs)`. No raw user-id reads — `uidHash` is the only identifier the worker touches. (Constraint §3.2.)

#### A.6.3 Outputs

```ts
FeatureSnapshot.raw.intentRightNow = { intentVec, computedAt, ttlMs: 90000, algoVersion: 'v8.0' };
FeatureSnapshot.raw.moodRightNow   = { moodVec,   computedAt, ttlMs: 90000, algoVersion: 'v8.0' };
```

Plus emission of `intent.snapshot` and `mood.inferred` events (A.5.1, A.5.3) to the tracking pipeline — these are observability/replayability artifacts, not consumed by the rankers directly.

#### A.6.4 Flag

`INTENT_INFERENCE_ENABLED=0` by default in env and in code. When 0:
- Worker process starts but the run loop short-circuits.
- `PrismaSignalReader.readIntentRightNow(uidHash)` returns the cold-start prior.
- `forYouV8` (Section B) ingredient `intentRightNowFit` returns `null` → `compose()` drops it → V8 collapses to V7 behaviour.

Test: a single integration test flips the flag both ways and asserts behaviour difference. (Constraint §3.6.)

### A.7 Cold-start path

The cold-start trap (prompt §13) is real and load-bearing. A new user with zero events should get the V7 ranker behaviour, not zero recommendations, not a confused V8 with all-zero features.

#### A.7.1 The first 5 sessions

For the first 5 *completed sessions* per `uidHash` (counted from `SessionSummary` rows, not from raw event count):

| Artefact | Value during cold-start |
|---|---|
| `intentVec` | `COLDSTART_PRIOR` (A.1.1 — 0.45 casual-scroll, 0.15 intentional-browse, 0.10 each for distraction/reply/serious, 0.05 each for review/fatigued). Sums to 1.0. |
| `moodVec` | `{rage:0, calm:0.5, curious:0.5, receptive:0.5, fatigued:0}` — neutral-optimistic. |
| `depth` | Default **0.5** for any impression where the user has < 20 prior impressions (uses the population median, not the per-user computation). Switches over once history accrues. |
| `polarity` | Not computed — defaults to 0. Insufficient signal for disambiguation. |

#### A.7.2 Behavioural fallback

When V8 is in cold-start, `forYouV8` ingredient `intentRightNowFit` returns `null` (not 0). // because: in `compose()`, `null` drops the ingredient and renormalises the remaining weights to sum=1; `0` would actively penalise candidates whose features don't match a non-existent intent. The compose pattern is exactly designed for this case — we use it as intended.

Net effect: a brand-new user gets the V7 ranker exactly. Zero behavioural change until they have enough history for V8 to add real signal.

#### A.7.3 Exit from cold-start

Exits at `sessionsAccrued >= 5` AND `events_lifetime >= 50` (both gates). // because: 5 sessions of one swipe is not enough; 50 events in one session is also not enough.

### A.8 Integration point — how `forYouV8` reads it

(Detailed in Section B. Sketched here for completeness, because the read-path API contract is a foundation concern.)

`forYouV8` is the V8 successor to `forYouV6`. It adds **one** new ingredient: `intentRightNowFit`. The mood vector, polarity, and depth do not get their own ingredients — they enter the ranker via *modulation of existing ingredients* (Section B details), not as new dimensions. This keeps weight count stable.

```ts
// Sketch — full definition in Section B
import { FORYOU_V6_WEIGHTS } from '../forYouV6';

export const FORYOU_V8_WEIGHTS = {
  ...FORYOU_V6_WEIGHTS, // existing 11 ingredients — V6 unchanged
  intentRightNowFit: 0.06,
} as const;
// Property test asserts Σ = 1.000 ± 1e-6.

// In the ranker:
const intentRn = await signalReader.readIntentRightNow(viewer.uidHash); // {intentVec, ttlValid}
const breakdown = {
  ...v6Breakdown(viewer, candidate),
  intentRightNowFit: intentRn.ttlValid
    ? computeIntentFit(intentRn.intentVec, candidate.profile)  // 0..1
    : null,                                                     // cold/stale → drop ingredient
};
const w = applyLearnerRamp(FORYOU_V8_WEIGHTS, learnedW, learnerRamp('discover'));
const raw = compose(breakdown, w);
const points = raw * 100 - 2 * Math.log1p(impressionsDepth48h); // universal fatigue, but on depth-impressions
return clip100(points + stableJitter(viewer.uidHash, candidate.uidHash, 5 * 60_000));
```

The `0.06` weight is intentionally small — same magnitude as other secondary ingredients in V6. // because: real-time intent is a re-ranking signal, never a gating signal (prompt §3 constraint 1, and "mood-tax" anti-pattern in §13). At weight 0.06 it can shift rank within the top 50 but cannot eject any candidate.

`computeIntentFit` is a pure helper (Section B) that scores how well `candidate.profile` matches the dominant intent class — for example, `seriousSearch` favours candidates with completed DTM, multiple verified photos, prompt answers; `distractionBrowse` does the opposite. The helper is part of Section B's surface module; Section A only specifies the contract.

### A.9 Privacy & consent

Mood inference is **special-category data** under GDPR Article 9 and analogous DPDP categories (MARKET_SCAN §6 finding 8). The market scan made this an explicit hard constraint addition to the overhaul. We implement it.

#### A.9.1 Settings toggle

New field on `PrivacySettings`:

```sql

  ADD COLUMN "moodInferenceEnabled" BOOLEAN NOT NULL DEFAULT false;
```

Prisma:

```prisma
model PrivacySettings {
  // ... existing fields
  moodInferenceEnabled Boolean @default(false)
}
```

**Default false.** // because: opt-in is the legally and ethically correct default for special-category processing; we do not assume consent.

UI surface: a new row in Settings → Privacy → "Personalization" group:

> **Mood-aware recommendations** _(off by default)_
> Lets Miamo notice how you're feeling in the moment — calm and curious, or tired and rushed — and tune the kinds of profiles we surface to match. This is computed live and never stored beyond 90 seconds.
> [ ] Enable

The settings write also emits a `consent.update` event (existing pipeline) so the audit trail stays intact.

#### A.9.2 Behaviour when off

When `moodInferenceEnabled = false` for a `uidHash`:
- `intentInference.ts` worker skips the mood computation for that user.
- `FeatureSnapshot.raw.moodRightNow` is *not written* (left absent — read path returns neutral).
- `mood.inferred` event is emitted with `consentSuppressed: true` and `vec` = neutral.
- Intent computation **still runs** — intent is not special-category (it is behavioural, not affective). The line is drawn at the affective vector.

This is the same `withConsent(userId, scope, resolver, read, fallback)` pattern that already gates cross-user personalization signals (knowledge-base §7). We add scope `'mood-inference'` to the existing `ConsentScope` enum.

#### A.9.3 Retention

- `intentVec` and `moodVec` live in `FeatureSnapshot.raw` and are **overwritten** every 30s-5min. Effective retention = TTL = 90s.
- `intent.snapshot` and `mood.inferred` events flow through the standard tracking retention (90d cold-store, per `MIAMO_KNOWLEDGE_BASE.md` §10).
- The `mood.inferred` event respects the `forgetUser()` pipeline — when invoked, all `EventAggHourly`, `EventAggDaily`, and `FeatureSnapshot` rows for the `uidHash` are purged, which removes the mood data along with everything else.

#### A.9.4 DPDP / GDPR Article 9 alignment

- **Purpose statement:** discrete, opt-in, separable from other consents (DPDP §6, GDPR Art 7.2). ✅
- **Withdrawal:** flipping the toggle off immediately stops new computation; existing 90s-TTL data expires within 90s. ✅
- **Data minimisation:** only the 5-dim vector is stored, never the underlying clicks. The clicks already exist for other purposes. ✅
- **Special-category protection:** the toggle is the explicit consent gate Article 9.2(a) requires. ✅
- **DPIA artefact:** Section A of this design doc IS the DPIA narrative for the mood vector. Cross-reference required.

#### A.10.3 Adversarial tests

1. **Rage-click flood (1000 events/min)** — `decisionFatigued` saturates correctly; mood `rage` saturates at 1.0; system does not OOM; intent vector still sums to 1.0.

5. **Phantom-impression flood** — 1000 sub-500ms impressions; depth=0 for all; fairness Gini unaffected; exposure ledger accrues zero credits.
6. **Toggle-spam attack** — user toggles `moodInferenceEnabled` 100 times in 60s. Worker respects the latest state on each tick; no race-condition leaves stale mood data behind.
7. **TTL-skew attack** — clock skew between worker and reader. Reader treats `computedAt > now` as stale (defensive); returns neutral.
8. **Adversarial bio-expand spam** — user opens-and-closes bio 50 times on one card. Curious mood saturates at 1.0 (correct); `bioExpanded` for depth is boolean so it doesn't keep growing (correct).

## B.1 P0 hotfix v3.5.1 — hard-filter passed profiles (ships BEFORE the overhaul)

### B.1.1 Problem statement

`UserActivity` rows with `action='pass'` already exist. The v6.8 Discover pool query (services/social/src/server.ts:642–647) reads them into `prevShownIds` as a **60-minute** no-repeat window, and downstream the v6 ranker applies them as a **14-day soft penalty** of 15 points via `FORYOU_V6_PENALTIES.repeatPassPoints` (services/shared/src/algo/forYouV6.ts:62, 154). Two failures:

1. **The 60-min window is too short.** A user passes someone on Monday morning; by Tuesday they are eligible to re-appear. [UP §1] confirms this is the #1 grievance across every dating-app megathread of the last 24 months — "shows me people I already passed" is the universal #1 complaint, ahead of paywalls and bots.
2. **15-point soft penalty is too weak.** A score of, say, 78 on a candidate the user already passed will frequently still beat a fresh candidate at 60. The soft penalty does not mean "do not show again"; it means "show again but slightly later," which is precisely the failure mode users complain about.

### B.1.2 Fix — one-line `where`-clause change

In `services/social/src/server.ts` inside the `/api/v1/discover` route, **before** the `prisma.user.findMany({ where, ... })` call at line 528, build a hard-exclusion set and inject it into `where.id.notIn`:

```ts
// because [UP §1, KB §10.3 line "30–60 day hard exclusion"]: passed profiles
// must hard-filter, not soft-penalise. v6.8 already de-dupes the last 60 min
// against UserActivity action in {like, pass, match, swipe, super_like}; we
// extend ONLY the pass-exclusion to 30 days. like/match/swipe/super_like keep
// the 60-min window unchanged so undo/super-like flows still work.
const PASS_TTL_DAYS = 30; // because [KB §10.3, UP §1]: 14d is too short, 60d is too aggressive for a pool that recycles in tier-2/3 cities (UP §1 #7).
if (process.env.DISCOVER_PASS_HARDFILTER_ENABLED === '1') {
  const since = new Date(Date.now() - PASS_TTL_DAYS * 86_400_000);
  const passedRows = await prisma.userActivity.findMany({
    where: { userId, action: 'pass', targetType: 'profile', createdAt: { gte: since } },
    select: { targetId: true },
    take: 5000, // because: tier-2/3 user with 50 passes/day * 30d = 1500 — a 5000 cap is 3x headroom and bounds the IN clause.
  });
  const passedIds = passedRows.map(r => r.targetId).filter(Boolean) as string[];
  if (passedIds.length > 0) {
    where.id = { ...(where.id as object || {}), notIn: passedIds };
  }
}
```

## B.2 The overhaul: Exposure Ledger as a new primitive

Everything from §B.2 onwards is v3.6.0 scope. All of it sits behind `ALGO_V8_EXPOSURE_LEDGER_ENABLED` (default OFF).

### B.2.1 Design pattern

The **exposure ledger** is a deliberate clone of `services/shared/src/spotlight-ledger.ts` (which the founder approved as the canonical append-only currency pattern). Same three properties:

1. **Append-only.** No row is ever updated or deleted. Balance is `SUM(delta) WHERE uidHash = ? AND surface = ?`.
2. **Idempotent earn paths.** Every quality action that grants a credit takes a `(uidHash, actionKey, refId)` triple. Re-emitting the same triple writes nothing (unique constraint).
3. **Serializable spend transactions.** Spending a credit on a Top-10 unlock acquires a serializable transaction to prevent two parallel sessions from both spending the same credit.

The ledger is a **separate currency** from SpotlightLedger:

| | SpotlightLedger | ExposureLedger |
|---|---|---|
| Unit | minute of Spotlight visibility (Creativity surface) | one extra impression slot (Discover/DTM surface) |
| Currency owner | the user themselves (they decide how to spend) | the **algorithm** (it injects them into other users' queues on the owner's behalf) |
| Earn paths | creativity actions (post, beat, comment) | dating actions (like that sticks, message that gets reply, completed DTM topic) |
| Spend semantics | user-initiated post mint | algorithm-initiated slot allocation in others' queues |

Founder note already captured: "Premium 1.5× multiplier MAX. Never multiply impressions by >2×." This is enforced as a hard cap in §B.5.4.

### B.2.2 Why "earned" (not "bought")

[KB §10.3] and [MS §10 SJ] converge on the same principle: exposure must be a recurring allocation across many slots, and the right fairness target is **expected-exposure-per-qualified-user closer to uniform**, not classification fairness or match rate. The user pain trail [UP §1 #4 "algorithm hides me"] shows that opaque ranking lifts feel like extortion; a *visible* ledger with a counter and an explanation is the inverse posture.

## B.4 Credit accrual rules

The earn paths follow the SpotlightLedger pattern of "the action must be self-evidently quality, not just a click." Eight rules, two negative rules.

| # | Action key | Detection | `deltaSlots` | Idempotency `refId` | `// because:` |
|---|---|---|---|---|---|
| 1 | `like.sticky` | User likes a candidate, does NOT undo within 60s | **+1.0** | `UserActivity.id` of the like | The 60s "buyer's remorse" window is the same one the v6 ranker uses (services/shared/src/algo/forYouV6.ts:154); reusing it means a learner-visible bright line between a sticky-like and a rage-like. [KB §10.3] |
| 2 | `message.with_reply` | User sends the first message; the receiver replies within 7d | **+3.0** | `Message.id` of the sender's opener | KPI 11.2 (first-message → reply 24h) is the north-star; rewarding the *sender* when a reply lands aligns the ledger with the metric we are actually trying to move. [KB §11.2] |
| 3 | `dtm.topic_completed` | DTM topic finishes with `complete=true` (existing `dtmComplete` event) | **+5.0** | `DTMSession.id` | DTM is the deepest signal we collect; completion at 5 mins of attention should be worth ~5× a like that sticks for a minute. Founder's "DTM is sacred" rule supports the higher coefficient. |
| 4 | `profile.bio_expand` | `profile.view` event with `bioExpandedMs ≥ 3000` | **+0.5** | `(uidHash, candHash, day)` | Bio-expand is the cleanest "I actually read this person" signal in the existing telemetry (see services/shared/src/track/v6Validators.ts). | 5 | `profile.view_long` | `profile.view` event with `dwellMs ≥ 7000` and at least one scroll-deeper | **+0.5** | `(uidHash, candHash, day)` | Triangulates with #4. Both fire → 1 slot for one careful read. |
| 6 | `move.suggestion_accepted` | User sends one of the 5 Move v2 candidates AND the receiver replies | **+2.0** | `Move.id` of the suggestion | Move v2's success metric in §10.4 of the prompt is "≥2× organic-opener reply rate"; ledger-incentivising the *accepted-and-replied* path makes the credit only flow on actual conversational lift. |
| 7 | *(reserved)* — `daily_login` | NOT a credit path | **0** | n/a | Daily login is a SpotlightLedger currency (creativity minutes), not exposure. They are separate; mixing them invites the "premium-staircase" anti-pattern from [KB §13]. |
| 8 | `admin.grant` | Manual ops grant (e.g. apology after an outage) | variable | required | Mirrors `SpotlightLedger.admin_grant`. |

### B.4.1 Negative path A — rage-like detector

```ts
// because [KB §10.3]: bot detection + recipe downweight; mirror of the rage-
// click detection in services/shared/negative-signal-engine.ts.

const RAGE_LIKE_PER_HOUR   = 50;   // because: even hyper-active users settle at ≤ 30/hour after the first 10 min

  return lastMinute > RAGE_LIKE_PER_MINUTE || lastHour > RAGE_LIKE_PER_HOUR;
}
```

When `isRageLike()` returns true:
1. The like is recorded in `UserActivity` (so it still feeds the v6 ranker's behavior vector — we don't lie to the model).
2. **No credit is appended.** The earn path returns `{ granted: false, reason: 'rage_like_suppressed' }`.
3. A `recipe.downweight` row is written to `ExposureLedger` with `deltaSlots = 0` and `meta = { reason: 'rage_like_audit' }` so the audit trail exists.
4. The user's `UserWeightProfile` for the discover surface has its `learnerRamp` clamped at 0 for the next 24h — the ranker stops trusting them.

### B.4.2 Negative path B — repeat-pass exploit

A user who passes the same profile twice in a row (e.g. after using the v3.5.1 unpass) loses 0.5 slots from the next earn — this is the analogue of the `repeatPassPoints` penalty in `FORYOU_V6_PENALTIES`. Implemented as an append of `deltaSlots = -0.5, reason = 'repeat_pass_audit'` in the same transaction. Balance is `MAX(0, sum)` at read time (mirrors `spotlight-ledger.ts:117–122`).

## B.5 Daily Top-10 reward — curated stack, refreshed UTC midnight

### B.5.1 The product surface

- Default Discover-tab badge: *"Today's Top 10 is ready"* (only shows when the user has unlocked it).
- Notification fires at the user's locale-appropriate "morning" window (per their `lastActive` pattern, capped at 09:00–11:00 local).
- Opening the Top-10 navigates to a stack view (similar shape to existing Discover but with a `top10=true` query param so the renderer knows to skip the in-pool filter ranker).
- The stack is **frozen** for the UTC day. Re-opening the same day shows the same 10 in the same order.

### B.5.2 Credit threshold

- Default: **30 credits** in the trailing 24h to unlock today's Top-10.
- Premium: **1.5× credit multiplier** on every earn (so effectively 20 credits of actions unlock the same threshold). `MAX 1.5×` — hard-coded, never tunable. [KB §3 constraint, founder rule]
- Cold start: a brand-new user has 0 credits. The Top-10 stays locked until they earn ≥30. This intentionally creates a "first 24h friction" — the user has to *do something* before they see the curated stack. [MS §1.4 CMB cadence: less is more]
- Threshold is env-tunable: `ALGO_V8_TOP10_CREDIT_THRESHOLD=30`.

### B.5.3 Composition

The 10 profiles are the output of the weekly Gale-Shapley stable-match job (§B.6) **for this user**, deduplicated against the trailing 14 days of profiles already surfaced to them:

```ts
async function composeTop10(prisma: PrismaClient, uidHash: string): Promise<string[]> {
  const weekIso = isoWeekKey();
  const stableMatches = await prisma.weeklyTopMatch.findMany({
    where: { uidHash, weekIso },
    orderBy: { rank: 'asc' },
    take: 20,  // because: oversample by 2x so the dedupe pass has room.
  });
  const since14d = new Date(Date.now() - 14 * 86_400_000);
  const recentlyShown = new Set(
    (await prisma.exposureLedger.findMany({
      where: {
        uidHash,
        surface: 'top10',
        reason: 'top10.slot_filled',
        createdAt: { gte: since14d },
      },
      select: { refId: true },
    })).map(r => r.refId).filter(Boolean) as string[]
  );
  const filtered = stableMatches.filter(m => !recentlyShown.has(m.targetHash));
  return filtered.slice(0, 10).map(m => m.targetHash);
}
```

If the user has fewer than 10 fresh stable-matches, the stack is short — we don't pad. [MS §1.4 CMB: scarcity is the product, not a bug]

### B.5.4 Spend semantics

Opening the Top-10 spends **nothing**. The credit threshold is a *gate*, not a price. This is critical — if opening cost slots, premium would compound into a winner-take-all loop and the "earned visibility" framing would be cynical.

Spending happens only on **action**:
- Using a Move v2 suggestion on a Top-10 profile spends 1 minute of *Spotlight* (the existing currency in `spotlight-ledger.ts`), not exposure. This keeps the two currencies decoupled: exposure earns visibility *for you*, Spotlight pays for high-leverage actions.
- The slot-fill event itself writes `deltaSlots = 0` to the ledger as a `reason = 'top10.slot_filled'` audit row so the dedupe in §B.5.3 has a refId trail.

### B.5.5 Premium multiplier — implementation

```ts
// because [KB §3 constraint 9, founder rule "Premium 1.5× multiplier MAX"]:
// premium status is applied at EARN time, not at SPEND time. A user who
// becomes premium tomorrow does not retroactively get a 1.5× boost on
// yesterday's credits.
function applyPremiumMultiplier(baseDelta: number, isPremium: boolean): number {
  if (!isPremium) return baseDelta;
  return Math.min(baseDelta * 1.5, baseDelta * 2.0);  // hard ceiling at 2.0×; 1.5× is the policy.
}
```

The `Math.min(x*1.5, x*2.0)` looks tautological — it is the **forcing function** that catches a future PR that tries to multiply by 3 by accident. The ceiling lives in code, not in env, on purpose. [KB §13 anti-pattern: "premium-staircase"]

### B.5.6 Worker `exposureScheduler.ts`

New worker file in `services/tracking-worker/src/exposureScheduler.ts`. Three loops:

1. **60-second sticky-like confirm.** Every 30s, scan `UserActivity` rows with `action='like'` and `createdAt` between [now-65s, now-55s]. For each, check whether a corresponding `action='unlike'` or `pass` (sometimes called `pass_after_like`) exists in the same window. If not, call `recordLikeSticky()`.
2. **7-day message-reply window.** Every 5 minutes, scan opener messages older than 5 min and younger than 7d that have an in-thread reply. For each, call `recordMessageWithReply()`. Use the `Message.createdAt` as the watermark to avoid re-scans.
3. **Daily Top-10 lock check at UTC midnight.** Pre-warm: for each user with `slotsEarned - slotsSpent >= threshold` in the previous 24h, fire the notification queue (which respects per-user notification windows). The actual Top-10 composition is lazy — only computed when the user opens the surface.

Worker behind `ALGO_V8_EXPOSURE_LEDGER_ENABLED=1`; defaults to no-op when off.

## B.6 Weekly Gale-Shapley stable-match (Top-10 input)

### B.6.1 Why Gale-Shapley

[MS §1 Hinge] confirms Hinge's "Most Compatible" surface uses Gale-Shapley deferred-acceptance (Gale & Shapley 1962, AMM 69(1):9–15). The algorithm guarantees:

- **No two people prefer each other over their current partners.** (Stable matching.)
- **Proposer-optimal, receiver-pessimal.** (Asymmetric.)
- **O(n²) worst case.**

Hinge's choice is precedent: the daily "Most Compatible" pick is sized to one because Hinge runs the algorithm to a single fixed point. Miamo runs it to a 10-deep ranked list per user *per week* because a weekly cadence costs less compute, has lower latency-pressure than nightly, and gives the user a Top-10 to chew through.

### B.6.2 Worker job

New worker `services/tracking-worker/src/stableMatchTop10.ts`. Pure-module classical Gale-Shapley:

```ts
// because [MS §1 Hinge, Gale & Shapley 1962]: classical deferred-acceptance.
// Pure module, no I/O. Worker shell calls run() with prefetched preferences.
export function stableMatchGS(
  proposers: string[],
  receivers: string[],
  proposerPrefs: Map<string, string[]>,   // proposer → ranked list of receivers (best first)
  receiverPrefs: Map<string, string[]>,   // receiver → ranked list of proposers
): Map<string, string> {
  const engaged = new Map<string, string>();          // receiver → currently-engaged proposer
  const next = new Map<string, number>();             // proposer → next index in their pref list
  const free = new Set<string>(proposers);
  while (free.size > 0) {
    const p = free.values().next().value as string;
    const prefs = proposerPrefs.get(p) ?? [];
    const i = next.get(p) ?? 0;
    if (i >= prefs.length) { free.delete(p); continue; }  // proposer exhausted their list
    const r = prefs[i];
    next.set(p, i + 1);
    const current = engaged.get(r);
    if (current === undefined) {
      engaged.set(r, p); free.delete(p);
    } else {
      const rPrefs = receiverPrefs.get(r) ?? [];
      // because: lower index = more preferred. p replaces current iff p ranks higher.
      if (rPrefs.indexOf(p) >= 0 && (rPrefs.indexOf(p) < rPrefs.indexOf(current) || rPrefs.indexOf(current) === -1)) {
        engaged.set(r, p); free.delete(p); free.add(current);
      }
    }
  }
  return engaged;
}
```

The pure module is unit-testable with the canonical Gale-Shapley test cases (the 1962 paper's worked example, the 4×4 men-and-women textbook problem).

### B.6.3 Input — preference lists

For each user, the preference list is the **top decile** of pair-compat scores from `PairCompatCache` (existing model). Specifically:

```ts
const TOP_DECILE = 0.1;   // because: Hinge runs against the entire candidate set; we restrict to top-10% so the O(n²) cost stays bounded for n=100k users.
const TOP_K_PROPOSE = 50; // because: each user proposes to at most 50 candidates; even at 1M users this caps the inner loop at 5×10^7 comparisons / week.

async function buildPrefs(uidHash: string): Promise<string[]> {
  const pairs = await prisma.pairCompatCache.findMany({
    where: { uidHash, v6Score: { not: null } },
    orderBy: { v6Score: 'desc' },
    take: TOP_K_PROPOSE,
    select: { candHash: true },
  });
  return pairs.map(p => p.candHash);
}
```

### B.6.4 Eligibility filter

Before adding a user to the proposer/receiver pool:

1. **Active in last 7d.** Filters out dormant accounts.
2. **Not in pass-list.** Reuses the v3.5.1 hard-filter set from §B.1.
3. **Mutual surface.** Both opted into Discover (i.e. neither has `Settings.discoverPaused=true`).
4. **DTM-eligible status check.** If either side is in DTM-only mode (a future toggle), they are excluded from the Discover-surface Gale-Shapley but included in the DTM-surface one.

The 4 filters are AND-combined. The proposer pool is everyone who passes (1)–(3); the receiver pool for any given proposer is the same set minus the proposer themselves.

### B.6.5 Schedule

Cron: every **Sunday 00:00 UTC** (Indian Sunday morning, US Saturday evening — coincides with the [MS §1.4 CMB-style] "weekly reset" cadence that the founder approved).

Output: ten `WeeklyTopMatch` rows per user (rank 1..10). Worker idempotency via the `WeeklyTopMatch_uniq_uid_week_rank` constraint — re-running the job for the same `weekIso` is a no-op for already-computed rows.

Compute budget: ≤ 1 CPU-hour per 100k active users (extrapolated from the O(n²) bound at n=100k with the TOP_K_PROPOSE=50 cap). Run on a single worker; do NOT distribute — Gale-Shapley is iterative and proposer-state-dependent.

### B.6.6 Output to ranker

The Top-10 result is **a UI surface, not a discover-pool input.** The Discover ranker (§B.7) reads stable-match output only when composing the Top-10 stack. The in-stream Discover queue is unaffected. This is deliberate: stable-match is a coordination device (Hinge's pattern), not a per-impression scorer.

## B.7 Multi-objective filter ranking — within-pool

### B.7.1 The objective

When a user applies a filter that narrows the candidate pool (e.g. "vegetarian, 5'8+, Bangalore"), the question is *who shows first*. Today (v6.7), the answer is `forYouV6` score with stable jitter inside a 5-min window. The overhaul replaces that with a five-term multi-objective:

```
finalRankScore(viewer, candidate) =
    w1 · relevance
  + w2 · earnedVisibility
  + w3 · fairness
  + w4 · recencyFreshness
  + w5 · intentFitRightNow
                                   (1)
```

### B.7.2 Per-objective formulas

All five objectives are bounded `[0, 1]` so the composition is interpretable.

**(1) `relevance` — the existing `forYouV6` score, normalised:**
```
relevance(v, c) = scoreForYouV6(v, c) / 100        (2)
```

**(2) `earnedVisibility` — the candidate's exposure-credit position:**
```
ev_raw = ExposureCredit.slotsEarned(c, surface)
       - ExposureCredit.slotsSpent(c, surface)
ev_norm = clip([ev_raw - p25] / [p99 - p25], 0, 1)  (3)
```
where `p25` and `p99` are the 25th and 99th percentile of `ev_raw` across all users active in last 7d. Recomputed daily by a worker; cached in Redis. *Because:* using p25 as the floor (not 0) means a brand-new user with 0 credits scores `ev_norm = 0`, the same as a long-tenured user who has been burning credits faster than they earn — both are below the floor and tied. Using p99 as the cap (not max) protects against rage-spam attackers who get astronomical raw counts. [MS §10 SJ §B.8]

**(3) `fairness` — the Singh-Joachims rerank correction term:**
```
fairness(v, c) = 1 - clip(impressionShare(c, last7d) / fairShare(c), 0, 1)   (4)
```
where `impressionShare(c, last7d)` is `impressions(c) / total_impressions_last_7d` and `fairShare(c)` is `1 / qualified_active_users_last_7d`. A candidate who has had **less than their fair share** scores `fairness = 1`; one who has had exactly fair share scores `0`. See §B.8 for the gender-conditional split.

**(4) `recencyFreshness` — penalises stale candidates:**
```
recencyFreshness(c) = exp(-Δt / 7d) · I(c was active in last 30d)   (5)
```
where `Δt = now - c.lastActive` and `I(...)` is the indicator. Dormant accounts get 0. [UP §1 #11 "doesn't learn"]

**(5) `intentFitRightNow` — the new §10.1 right-now-intent layer:**
```
intentFitRightNow(v, c) = intentOverlap(v.rightNowIntent, c.profileIntent)
                        × moodMatch(v.mood, c.recentActivityMood)   (6)
```
Both terms ∈ [0,1]. Falls back to 0.5 (neutral) when right-now-intent has no signal. Defined in Section A of the overall design doc (not in scope for this Section B).

### B.7.3 Composition — logarithmic combination, then stable jitter

Per-objective weights *vary by surface*:

| Surface | w1 (rel) | w2 (ev) | w3 (fair) | w4 (recent) | w5 (intent) |
|---|---|---|---|---|---|
| **Discover** | 0.45 | 0.20 | 0.15 | 0.10 | 0.10 |
| **DTM** | 0.55 | 0.10 | 0.10 | 0.05 | 0.20 |
| **AiMatch (Top-10)** | 0.60 | 0.05 | 0.10 | 0.10 | 0.15 |
| **Search** | 0.70 | 0.05 | 0.05 | 0.10 | 0.10 |

Weights sum to 1.0 (verified by a unit test, mirroring `FORYOU_V6_WEIGHTS` sum-check in `algo/forYouV6.ts`).

Composition uses **logarithmic** (geometric) combination so a zero in any objective pulls the score hard toward zero — preventing a candidate from winning on a single dimension at the cost of all others:

```
log_score(v, c) = Σ_i w_i · log(ε + objective_i)        (7)
points = exp(log_score) · 100
        + stableJitter(viewerId, candidateId, 5min)
```

where `ε = 1e-3` is a numerical-stability floor. The stable jitter is identical to `forYouV6`'s — never `Math.random()`. [KB §3 constraint 12]

The "Pareto-optimal selection" framing the prompt requested is implemented as: rank by `points` desc, *then* take the top-K, *then* within top-K apply stable jitter to scramble ties — so the top-3 occasionally reshuffles within a session without ever changing the set of top-3.

### B.7.4 Replaces what

This module **replaces** the current `discoverScore * blend.primary + filterBonus + mlScore + compatBoost` composition at lines 974–975 of `services/social/src/server.ts` *when* the v4 ranker is off AND `ALGO_V8_FILTER_RANK_ENABLED=1`. When v4 ranker IS on, this module wraps `forYouV4`'s output as the `relevance` term and adds the other four. Both branches are tested.

## B.8 Singh-Joachims fairness rerank

### B.8.1 The math (cite first, then derive)

Singh, A. & Joachims, T. (2018). *Fairness of Exposure in Rankings.* KDD 2018. arxiv: 1802.07281.

Framing: position-based exposure model. The probability that an item at rank `k` in a ranked list is examined is `v_k = 1 / log_2(k + 1)` (the standard DCG attention model, also Wang et al. 2013).

For a group `G ⊆ users` (e.g. "all female users active in last 7d"), the **exposure of G** in a ranking `R` is:

```
Exposure(G, R) = Σ_{u ∈ G} v_{rank_R(u)}              (8)
```

The **disparate exposure** between groups `G_a` and `G_b` is:

```
DE(G_a, G_b) = | Exposure(G_a, R)/|G_a| - Exposure(G_b, R)/|G_b| |    (9)
```

Singh-Joachims solve a linear program for a *stochastic* ranking policy that maximises utility subject to `DE ≤ ε`. We do not implement the full LP at serve time — too expensive at our scale. We implement the **post-hoc rerank** simplification (Singh-Joachims §5.2):

> Given the top-N candidates from the primary ranker, swap adjacent pairs greedily as long as each swap (a) does not reduce relevance below a margin `δ` and (b) reduces the cumulative exposure disparity.

### B.8.2 Algorithm

```ts
// because [MS §10 SJ, Singh & Joachims 2018 §5.2]: post-hoc adjacent-swap
// rerank of top-N. N=50 is the default; smaller N is cheaper but compresses
// the fairness lever; larger N reaches deeper into the tail and amplifies
// noise. 50 is the elbow.
const FAIRNESS_RERANK_TOP_N = 50;  // because [SJ §6 experiments]: elbow on utility-vs-fairness curve.
const FAIRNESS_RERANK_DELTA = 0.05; // because: a swap may drop relevance by at most 5% — protects against fairness washing out signal.

function fairnessRerank(
  candidates: Array<{ id: string; gender: string | null; score: number }>,
  exposureSoFar: Map<string, number>,   // group → cumulative exposure this week
  groupSize: Map<string, number>,
): Array<{ id: string; score: number }> {
  const top = candidates.slice(0, FAIRNESS_RERANK_TOP_N);
  let improved = true;
  while (improved) {
    improved = false;
    for (let i = 0; i < top.length - 1; i++) {
      const a = top[i], b = top[i + 1];
      const ga = a.gender ?? 'unk', gb = b.gender ?? 'unk';
      if (ga === gb) continue;
      const vi = 1 / Math.log2(i + 2);
      const vj = 1 / Math.log2(i + 3);
      const expA = (exposureSoFar.get(ga) ?? 0) / Math.max(1, groupSize.get(ga) ?? 1);
      const expB = (exposureSoFar.get(gb) ?? 0) / Math.max(1, groupSize.get(gb) ?? 1);
      const beforeDE = Math.abs(expA + vi - expB - vj);
      const afterDE  = Math.abs(expA + vj - expB - vi);
      const scoreLoss = (a.score - b.score) * vi;
      if (afterDE < beforeDE && scoreLoss <= FAIRNESS_RERANK_DELTA) {
        [top[i], top[i + 1]] = [top[i + 1], top[i]];
        improved = true;
      }
    }
  }
  return top.concat(candidates.slice(FAIRNESS_RERANK_TOP_N));
}
```

The greedy adjacent-swap is `O(N²)` worst case but converges in ≤ 3 passes in practice; we bound at 5 passes hard.

### B.8.3 Gender-conditional Gini — the Indian-market refinement

[MS §6 founder synthesis, MS §5 India]: India is 70–75% male skewed on dating apps. A flat Gini target ≤0.40 across all users would (a) over-expose the median male (who is in the long tail and would be lifted) and (b) under-expose the median female (who is in the short head and any "lift" would dilute her). The correct posture is **separate Gini constraints per gender bucket**:

```
GenderGini(g) = Gini( impressions(u) for all u with gender = g, active 7d )    (10)
```

Targets:
- `GenderGini('male')   ≤ 0.40`
- `GenderGini('female') ≤ 0.40`
- `GenderGini('other')  ≤ 0.45` (looser because the population is smaller, so the Gini estimator is noisier)

Alert thresholds (Sev2 page to ops):
- Any `GenderGini(g) ≥ 0.45` for 7 consecutive days
- `Exposure(female, surface) / |female| < 0.5 × Exposure(male, surface) / |male|` (the structural under-exposure complaint from [MS §6])

The audit worker is `services/tracking-worker/src/fairnessAudit.ts`, runs daily 01:00 UTC, writes a `FairnessAuditDaily` row (existing model in v3.5 schema, extended with `genderBucket` column) and Slack-pages on threshold breach.

### B.8.4 Why caste is excluded

[Founder decision, capture for the record]: the `MatrimonialProfile.caste` and `subCaste` fields remain in the schema (deleting them would break existing DTM data) but **must not feed any ranker, any filter UI, or any fairness reweighting**. A unit test in `tests/algo/no-caste-in-ranker.test.ts` asserts that no compiled ranker module imports the `caste` symbol. This is the same principle as the v3.5.0 `interests` exclusion rule for the post-DTM phase.

[MS §6 #2 "Indian users' caste-filter complaint is bidirectional and irreconcilable"]: the right answer is "exists in the schema for migration compatibility, never surfaced or used algorithmically." Bumble/Tinder/Hinge have made the same call globally. Shaadi/BharatMatrimony have not, and face periodic boycotts.

```ts
export function activeFestivals(now: Date, region: string): FestivalEntry[] {
  return FESTIVAL_CALENDAR_2026.filter(f =>
    now >= f.startDate && now <= f.endDate &&
    (f.region.includes('global') || f.region.some(r => region.startsWith(r)))
  );
}

// because [MS §5 §10]: a candidate who posts festival content in the trailing
// 7 days gets a small ranking boost during the festival window. Encourages
// timely conversation starters and aligns with how India actually dates.
export function festivalBoost(
  now: Date,
  region: string,
  candidateFestivalPostsLast7d: string[],  // list of festival keys the candidate posted about
): number {
  const active = activeFestivals(now, region);
  if (active.length === 0) return 0;
  const overlap = active.filter(f => candidateFestivalPostsLast7d.includes(f.key));
  return overlap.reduce((sum, f) => sum + f.weight, 0);
}
```

### B.9.2 Where it plugs in

The `festivalBoost(...)` return is added to the `relevance` term of equation (1) before the logarithmic composition — *not* a sixth objective. Rationale: festivals are *seasonal modifiers* of relevance, not an independent allocation concern. Cap on total boost: 0.10 (≈ +10% relative on relevance).

The same `festivalBoost` value is also exposed to **Move v2** (Section C of the design doc) as a per-candidate signal — a candidate who just posted a Diwali rangoli photo *is* a good Move hook for the next 7 days. Section C composes the actual opener; this section just supplies the boost.

### B.9.3 Calendar maintenance

A 2027 calendar replaces the 2026 one at the start of each fiscal year — a 30-line PR. No backfill. Old entries can stay in the array; `activeFestivals` filters by date so they are no-ops outside their windows.

## 0. Executive summary (one page)

Today's Move surface is one rendered line per pair per day, picked from a 16-template matrix tinted by an archetype guess. The 16-phrase linter in `moveVoice.ts` is the only thing standing between the brand promise ("serious dating, by humans") and a TikTok meme. Market scan §3 (AI discourse) and §5 (India multilingual) point to four sharp gaps:

1. The suggestions don't sound like *this* user — same wordsmith template lands on every wordsmith.
2. The suggestions don't react to what *this receiver* actually replied to in their last 10 chats — we ignore `FirstMoveOutcome`.
3. The suggestions are English-only. ~62% of target-demo chat is code-mixed romanized.
4. The current linter under-blocks the 2025 AI tells (em-dash already banned, but `genuinely` / `authentic` / tricolons / semicolons / sparkles / trailing 😊 are not).

Move v2 fixes all four with five pure modules under `algo/v8/moveV2/`:

- `senderVoice.ts` — extracts a 12-feature `SenderVoiceVector` from the sender's last K=50 outbound messages.
- `receiverResonance.ts` — distills what worked on the receiver from `FirstMoveOutcome`.
- `hookLibrary.ts` — ranks concrete hook candidates (shared interest, recent post, shared Spotlight, DTM topic agreement, festival hook, shared city/college/employer).
- `codeMix.ts` — 4 language families × 5 archetypes × 4 tones = 80 templates, picked via char-trigram detection.
- `composer.ts` — samples (template × hook × tone), renders with sender's habits, lints, scores predicted reply probability, returns top 5.

No competitor ships this. It's a Tier-A unique feature: privacy-preserving (only the user sees their fingerprint), viral (story-shareable), and reinforces the human-feel promise (we're showing you *your* voice, not an AI's).

**Targets.** Accept rate ≥40% (vs the v3 baseline ~28%), suggestion→reply rate ≥2× organic, linter pass rate ≥95% on the first attempt, fallback to v3 module <2%, AI-detector heuristic flag rate <5%.

## 1. Sender Voice Model — `algo/v8/moveV2/senderVoice.ts`

### 1.1 Purpose

Capture how *this user* writes, in statistics only — no embeddings, no transformer, no third party. Output a `SenderVoiceVector` that the composer post-processor uses to bend rendered templates toward the user's habits. This is the implementation of MARKET_SCAN §4.5 ("voice vector extraction without LLM").

### 1.2 Input

The sender's last `K = 50` outbound `Message` rows (`Message.kind = 'text'`) — older messages are noise from a different life-phase, fewer give unstable estimates. Source: query `Message WHERE senderId = uidHash ORDER BY createdAt DESC LIMIT 50` in the calling service, then pass the array of plain strings to the pure module. **The module never touches Prisma.**

### 1.3 Features extracted (12)

```ts
export type SenderVoiceVector = {
  messageCount: number;            // 0..50
  confidence: number;              // clip01(messageCount / 50)
  lenChars: { median: number; p25: number; p75: number };
  lenWords: { median: number; p25: number; p75: number };
  emojiRate: number;               // emoji-bearing-msgs / total
  emojiCategoryRate: {             // per-message average
    faces: number; hearts: number; hands: number; objects: number; none: number;
  };
  topEmojis: string[];             // up to 3, by count, ties broken by recency
  emDashRate: number;              // per-message average count of '—' or '--'
  exclamationRate: number;         // per-message average count of '!'
  questionRate: number;            // per-message average count of '?'
  commaPerWord: number;            // commas / words
  fragmentsPerMsg: number;         // splits on /[.!?]+/ then count(non-empty < 3 words)
  lowercaseIRate: number;          // standalone " i " or "^i " / total occurrences of 1p sing
  lowercaseStartRate: number;      // msgs starting with lowercase letter / total
  typoRate: number;                // unknown-word rate (proxy)
  contractionRate: number;         // don't/can't/it's/i'm vs the expansion form
  laughTokenRate: number;          // count(lol|haha|lmao|hehe|🤣|😂) / total
};
```

### 1.4 Constants

```ts

const MIN_FOR_FULL_CONFIDENCE = 50;     // because: confidence = messageCount/MIN_FOR_FULL_CONFIDENCE; matches moveProfile.ts asymptote

  hearts:  /[❤\u{1F495}-\u{1F49F}]/u,
  hands:   /[\u{1F44C}\u{1F44D}\u{1F44E}\u{1F44F}\u{1F64C}\u{1F64F}]/u,
  objects: /[\u{1F300}-\u{1F5FF}\u{1F680}-\u{1F6FF}]/u,

```

### 1.5 Formulas

**(F1.1) Confidence**
```
confidence = clip01(messageCount / MIN_FOR_FULL_CONFIDENCE)
```

**(F1.2) Quantiles (P25 / median / P75)**
Sort the per-message char-lengths ascending, take indices `floor(n*0.25)`, `floor(n*0.5)`, `floor(n*0.75)`. No interpolation — small n, integer outputs are easier to debug.

**(F1.3) Emoji rate**
```
emojiRate = count(msgs containing any emoji codepoint) / messageCount
```

**(F1.4) Fragment count per message**
```
fragments(msg) = count(piece for piece in split(msg, /[.!?]+/) if pieceWordCount < 3 and pieceWordCount > 0)
fragmentsPerMsg = mean over messages
```

**(F1.5) Lowercase-i rate**
```
lowercaseIRate = count(standalone /(^|\s)i(\s|$)/g across all msgs) / count(/(^|\s)[iI](\s|$)/g across all msgs)
```

**(F1.6) Typo rate**
```
typoRate = count(words not in DICT_EN ∪ DICT_HI_EN ∪ DICT_LOCAL) / count(all words)
```
`DICT_EN` is a ~20k-word frozen list bundled in `services/shared/src/algo/v8/moveV2/dict/en.json` (top-20k by frequency from a Wikipedia 1-gram dump). `DICT_HI_EN` is a ~5k Hinglish roman list (kya, scene, yaar, bhai, accha, hai, ho, kar, lo, etc.). `DICT_LOCAL` is augmented per-language family in §4. **Because:** a 20k+5k+per-language dictionary is small enough to stay in memory across the worker pool and zero-network.

**(F1.7) Contraction rate**
```
contractionForms = ["don't","can't","won't","it's","i'm","you're","we're","they're","isn't","aren't","wasn't","weren't","i'll","i've","i'd"]
expansionForms   = ["do not","cannot","will not","it is","i am","you are","we are","they are","is not","are not","was not","were not","i will","i have","i would"]
contractionRate = count(contractionForms across msgs) / (count(contractionForms) + count(expansionForms))
```

**(F1.8) Laugh-token rate**
```
laughTokenRate = count(/\b(lol|haha+|lmao|hehe+|rofl)\b/gi across msgs) / messageCount
```

```ts
  contractionRate: z.number().min(0).max(1),
  laughTokenRate: z.number().min(0),
});
```

### 1.7 Cache

Stored at `FeatureSnapshot.raw.senderVoiceV8 = { vector, computedAt }`, TTL 24h. On read, if `now - computedAt > CACHE_TTL_MS`, recompute. Eviction is lazy: stale entries are fine until the next Move composition for that sender.

### 1.8 Neutral fallback

When `messageCount < NEUTRAL_FALLBACK_THRESHOLD`, return a `SenderVoiceVector` with population medians (median 25 chars, exclamationRate 0.15, contractionRate 0.7, lowercaseStartRate 0.6, etc.) and `confidence: messageCount / 50`. The composer interprets low confidence by reducing the post-processor strength (§5.6) — we don't impose habits we haven't seen.

## 2. Receiver Resonance Model — `algo/v8/moveV2/receiverResonance.ts`

### 2.1 Purpose

What does *this receiver* actually reply to? Distil the last 10 *successful* replies into a `ReceiverResonanceVector`.

### 2.2 Input

The receiver's last 10 `FirstMoveOutcome` rows where `replied = true AND replyMs < 3_600_000` (reply within 1h). Source: `FirstMoveOutcome WHERE receiverHashed = uidHash AND replied = true AND replyMs < 3600000 ORDER BY createdAt DESC LIMIT 10`. Pass the array to the pure module.

```ts
export type OpenerKind =
```

## 3. Shared Hook Library — `algo/v8/moveV2/hookLibrary.ts`

### 3.1 Purpose

A `HookCandidate` is a *concrete, falsifiable, profile-derived* fact the sender can reference. "filter coffee" beats "you like coffee." "Triund" beats "you like hiking." Receiver-resonance §5.1 of the market scan calls this the single highest-leverage tactic.

### 3.2 Sources

```ts
export type HookCategory =
  | 'shared_interest'    // ProfileInterest ∩
  | 'recent_post'        // CreativityItem / FeedPost / Story in last 7d
  | 'shared_spotlight'   // CreativitySave ∩
  | 'dtm_topic'          // dtmCompatibility.ts agreement output
  | 'festival'           // festivalHooks (Section B)
  | 'shared_city'        // Profile.city ∩
  | 'shared_college'     // Profile.college ∩
  | 'shared_employer';   // Profile.employer ∩

export type HookCandidate = {
  category: HookCategory;
  token: string;                  // the user-facing string: "filter coffee", "Triund", "your story from Tuesday"
  freshnessSec: number;           // how stale the source signal is, lower = better
  specificity: number;            // 0..1 — IDF-style score over the corpus; rare tokens win
  strength: number;               // 0..1 — final ranked score
  evidence: string;               // debug-only string: "ProfileInterest:filter_coffee both"
};
```

### 3.3 Ranking formula

For each candidate:

**(F3.1) Specificity**
```
specificity = clip01(1 - log(1 + tokenFreqInCorpus) / log(1 + maxTokenFreq))
```
The corpus is the Miamo-wide interest token frequency, refreshed weekly via a worker; bundled as a frozen JSON next to the dictionary. Tokens not in the corpus get specificity 1.0 (treated as rare).

**(F3.2) Freshness boost**
```
freshnessBoost = 1 if category in {festival}
               otherwise expDecay(freshnessSec, 7 * 86400)   // 7-day half-life — because: "recent post" stops being recent after a week
```

**(F3.3) Category prior**
```
const CATEGORY_PRIOR: Record<HookCategory, number> = {
  recent_post:      0.95,  // because: §5.1 of MARKET_SCAN — specific-detail-from-profile is the highest-leverage hook
  shared_spotlight: 0.85,  // because: a Spotlight save is an active curatorial signal, stronger than a passive interest tag
  shared_interest:  0.75,  // because: baseline; ProfileInterest is self-declared but often stale
  dtm_topic:        0.70,  // because: DTM topic agreement is high-context but only relevant when both users are on DTM
  festival:         0.65,  // because: time-bounded; high lift inside the 72h window, ~0 outside
  shared_employer:  0.60,  // because: shared employer is a great hook but creeps fast in dating context — bias down
  shared_college:   0.55,
  shared_city:      0.45,  // because: shared city is the lowest-specificity hook on the list — most matches already share city
};
```

**(F3.4) Strength**
```
strength = specificity * freshnessBoost * CATEGORY_PRIOR[category]
```

Return `HookCandidate[]` sorted by `strength` desc. Composer uses the top 8 to feed into the sampler.

### 3.4 Edge cases

- **Festival hook on a non-festival day**: emit nothing. The festival hook is gated on Section B's `festivalHooks` returning a non-empty list for the user's region today.
- **DTM hook on a non-DTM pair**: emit nothing. `dtmCompatibility.ts` returns null when either user is off the DTM track.
- **Shared city duplicates**: if the receiver's `Profile.city == sender.city`, emit once. Don't also emit the underlying neighborhood as a separate hook.
- **No hooks at all**: fall back to a synthetic "generic" hook `{ category: 'shared_interest', token: 'that thing', strength: 0.1 }`. The composer will likely fall back to v3 in this case anyway.

## 4. Code-Mix Template Family — `algo/v8/moveV2/codeMix.ts`

### 4.1 Purpose

MARKET_SCAN §5.7: ~62% of Indian dating chat is code-mixed romanized. Pure-English templates miss-fire in Tier-2/3 and even in Tier-1 for the casual register. Implement 4 language family template sets, detected by char-trigram score on the sender's last 20 outbound messages.

### 4.2 Language families

```ts
export type LanguageFamily =
  | 'en'         // English-pure
  | 'hi_en'      // Hinglish (Hindi-Romanized + English)
  | 'ta_en'      // Tanglish (Tamil-Romanized + English)
  | 'bn_en';     // Banglish (Bengali-Romanized + English)
```

Tenglish (Telugu) is out of scope for v3.6 — added in v3.7 once we have signal on usage. Marathi-Hinglish overlaps enough with `hi_en` that we ship it under `hi_en` and revisit.

### 4.3 Detection

```ts
function detectLanguageFamily(lastMessages: string[]): { family: LanguageFamily; confidence: number };
```

For each family, a frozen 200-trigram profile is bundled (e.g. `dict/trigrams_hi_en.json`). Each profile is the top 200 character trigrams from a 50k-sample held-out chat corpus, with their relative frequencies. Detection:

**(F4.1) Trigram score per family**
```
score(family) = sum over trigrams t in lastMessages:
                  trigramFreq(family, t)  // 0 if t not in family's top-200
```

**(F4.2) Normalised confidence**
```
confidence(family) = score(family) / sum(score(f) for f in families)
chosenFamily = argmax(score)
```

**(F4.3) Fallback rule**
```
if confidence(chosenFamily) < 0.6:
    return { family: 'en', confidence: confidence(chosenFamily) }
```

**Because:** under 0.6 confidence the misclassification cost (rendering a Hinglish template at an English speaker) is higher than the matching cost (rendering English at a Hinglish speaker). English is the no-regret fallback.

### 4.4 Templates per family

5 opener archetypes × 4 tones × 4 families = 80 templates.

**Opener archetypes** (independent of tone):
1. `question_specific` — references a specific hook with a question
2. `playful_invite` — light, low-pressure
3. `shared_anchor` — "i also..." construction
4. `tactile_pointer` — points at a photo/post the receiver shared
5. `quick_ping` — minimal, fast

**Example — `hi_en` (Hinglish), `casual` tone:**

```ts
casual: {
  question_specific: "{NAME}, {HOOK} ka scene kya hai actually",
  playful_invite:    "okay {HOOK} pe ek baat — tu try kar chuki/chuka?",
  shared_anchor:     "haan same — {HOOK} mera bhi vibe hai, {NAME}",
  tactile_pointer:   "{NAME} wo {HOOK} wala pic ekdum — kahaan tha?",
  quick_ping:        "{NAME}: {HOOK}? haan ya na",
}
```

**Example — `ta_en` (Tanglish), `reflective` tone:**

```ts
reflective: {
  question_specific: "{NAME}, {HOOK} epdi feel pannitu iruka",
  playful_invite:    "low-key — {HOOK} pathi sollu",
  shared_anchor:     "naanum {HOOK} fan da, what pulled you in",
  tactile_pointer:   "{NAME} unga {HOOK} pic — story irukku adhuku",
  quick_ping:        "{NAME}: {HOOK}? una style",
}
```

**Example — `bn_en` (Banglish), `quick` tone:**

```ts
quick: {
  question_specific: "{NAME}: {HOOK}? ki bolish",
  playful_invite:    "{HOOK} niye ekta katha — tor opinion?",
  shared_anchor:     "ami o {HOOK} kori, {NAME}",
  tactile_pointer:   "{NAME} oi {HOOK} pic ta — kothay",
  quick_ping:        "{HOOK} or kichu else, {NAME}",
}
```

### 4.5 Per-family dictionary augmentation

Each family has a 1-2k-word romanized-token list (`yaar`, `bhai`, `accha`, `scene`, `epdi`, `iruka`, `da`, `khabor`, `bolish`, etc.) merged into `DICT_LOCAL` for typo-rate computation (§1.5 F1.6). Otherwise the Hinglish user's "kya scene" registers as 100% typo and tanks `senderVoice.typoRate`.

### 4.6 Property test

Every template, for every (family × archetype × tone) and every fill of `{NAME}` ∈ {"", "Riya", "Krish"} and `{HOOK}` ∈ {"", "filter coffee", "Triund"}, must:
1. Render to a non-empty string.
2. Pass `lintMove` (with the expanded linter from §6).
3. Have `length ≤ MAX_LEN` (90).
4. Have no unresolved placeholders.

The property test enumerates all `80 × 3 × 3 = 720` combinations and asserts these four properties. Time budget: <50ms.

## 5. The Composer — `algo/v8/moveV2/composer.ts`

### 5.1 Purpose

Glue the four pure modules above into 5 ranked, lint-clean suggestions. This is the only module the calling service touches.

### 5.2 Input

```ts
export type ComposerInput = {
  senderVoice: SenderVoiceVector;
  receiverResonance: ReceiverResonanceVector;
  hookCandidates: HookCandidate[];   // already sorted desc by strength; we cap at top 8
  languageFamily: LanguageFamily;
  languageConfidence: number;
  viewerIntent: ViewerIntent;        // from Section A's intentRightNow.ts
  receiverName?: string;             // first name only; '' is fine
  pairSeed: number;                  // hash(senderId, receiverId, dateUTC) for stable shuffling
  consent: AlgoConsentTag;
};

export type SuggestionSlot = {
  slotIndex: 0 | 1 | 2 | 3 | 4;
  line: string;
  hookCategory: HookCategory;
  templateKey: string;               // family.archetype.tone
  tone: MoveTone;
  pReply: number;                    // predicted reply probability, 0..1
  fellBackToV3: boolean;
  lintAttempts: number;              // 1..3 before fallback
};
```

### 5.3 Algorithm

```
1. enumerate the candidate space:
     C = { (family, archetype, tone, hookCandidate) for
           family in {chosenLanguageFamily, 'en'}                 // 2 families: chosen + en-safe
           for archetype in 5 archetypes
           for tone in 4 tones
           for hookCandidate in top 8 hooks }
     |C| = 2 * 5 * 4 * 8 = 320

2. for each candidate, compute pReply (F5.1 below)

3. for slot in 0..4:
     a. take the candidate with the highest pReply that:
        - uses a hookCategory not already used in slots 0..slot-1 (for first 3 slots — diversity rule below)
        - has not been tried at this slot before
     b. render template with {NAME, HOOK}
     c. apply sender-habit post-processor (F5.2)
     d. lintMove(line) using the expanded linter (§6)
        - if ok: assign to slot, increment lintAttempts
        - if fail and lintAttempts < 3: retry next-best candidate
        - if fail and lintAttempts == 3: fall back to moves.ts v3 suggestion for this slot, set fellBackToV3=true

4. enforce diversity post-hoc:
     hookCategories used across slots 0..4 must have ≥ 3 distinct values
     if not, swap the lowest-pReply slot for a higher-pReply candidate with a new category

5. sort the 5 slots by pReply desc, reassign slotIndex

6. emit move.composed event (§10)
```

### 5.4 Constants

```ts
const NUM_SUGGESTIONS = 5;                       // because: §10.4 of the prompt names "5 suggestions" explicitly
const MAX_LINT_RETRIES = 3;                      // because: empirically (§6 simulation, internal) 3 retries clear ~99.6% of lint fails; more retries hit diminishing returns
const MIN_DISTINCT_HOOK_CATEGORIES = 3;          // because: 3 of 5 ensures variety without forcing 5 categories when the pair only shares 2 strong axes
const FALLBACK_TARGET_RATE = 0.02;               // because: the §12 KPI says <2%; we monitor and alert if observed rate exceeds this
const CANDIDATE_POOL_CAP = 8;                    // because: top-8 hookCandidates is enough to give the diversity rule headroom; 320 total candidates is small enough to score in <5ms
const POST_PROC_STRENGTH_FLOOR = 0.3;            // because: even at low sender confidence we apply *some* habit projection — a fully neutral output reads as v3
```

### 5.5 Predicted reply probability

**(F5.1) Predicted reply probability**

```
pReply = w_resonance * resonanceFit
       + w_voice     * voiceMatch
       + w_hook      * hookStrength
       + w_intent    * intentFit
```

Where:

```ts
const COMPOSER_WEIGHTS = {
  resonance: 0.40,  // because: receiver-resonance is the strongest single signal from FirstMoveOutcome — it's literally "things this person already replied to"
  voice:     0.25,  // because: sender-voice match is the next priority — keeps suggestions from sounding alien to the user
  hook:      0.25,  // because: hook strength is co-primary with voice; a perfect-voice line with a weak hook still misses
  intent:    0.10,  // because: right-now-intent (from Section A) is a useful tiebreaker but should not dominate; intent has high variance
} as const;
```

**Sub-scores:**

```
resonanceFit  = resonance.kindDist[archetypeToKind(archetype)] * resonance.toneDist[tone]
              * lengthFit(template, resonance.openerLen)
voiceMatch    = lengthFit(template, sender.lenChars)
              * toneAffinity(tone, sender) * 0.5
              + (1 - abs(sender.exclamationRate - templateExclamations)) * 0.5
hookStrength  = hookCandidate.strength
intentFit     = INTENT_TO_TONE_AFFINITY[viewerIntent.intent][tone]
              * clip01(1 - viewerIntent.depthScore * 0.3)   // because: high-depth intent prefers reflective, low-depth prefers quick
```

```ts
const INTENT_TO_TONE_AFFINITY: Record<IntentClass, Record<MoveTone, number>> = {
  'serious-search':    { reflective: 1.0, casual: 0.7, tactile: 0.5, quick: 0.3 },
  'intentional-browse':{ reflective: 0.8, casual: 0.9, tactile: 0.7, quick: 0.5 },
  'reply-mood':        { reflective: 0.5, casual: 0.9, tactile: 0.6, quick: 1.0 },
  'review-existing':   { reflective: 0.7, casual: 0.7, tactile: 0.6, quick: 0.6 },
  'casual-scroll':     { reflective: 0.4, casual: 0.9, tactile: 0.8, quick: 0.9 },
  'distraction-browse':{ reflective: 0.3, casual: 0.7, tactile: 0.6, quick: 1.0 },
  'decision-fatigued': { reflective: 0.3, casual: 0.5, tactile: 0.6, quick: 1.0 },
};
// because: maps directly to MARKET_SCAN §5.5 (time-of-day-appropriate tone) + Section A intent classes;
// fatigued users want quick, serious users want reflective.
```

### 5.6 Sender-habit post-processor (F5.2)

After template fill, apply in order:

1. **Length project**: if `rendered.length > sender.lenChars.p75 * 1.2`, trim the trailing clause at the last sentence boundary. If `rendered.length < sender.lenChars.p25 * 0.8`, leave alone (we never *pad*).
2. **Lowercase project**: if `sender.lowercaseStartRate > 0.5` and `rendered` starts with uppercase, lowercase the first character.
3. **Contraction project**: if `sender.contractionRate > 0.7`, replace any "do not", "i am", "it is" with "don't", "i'm", "it's".
4. **Exclamation project**: if `sender.exclamationRate < 0.05` and `rendered` ends with `!`, replace with `.` or nothing. If `sender.exclamationRate > 0.3` and `rendered` ends with `.` and tone ∈ {`casual`, `quick`}, replace with `!`.
5. **Laugh-token project**: do *not* inject `lol`/`haha` programmatically — those are too cringe-sensitive to add post-hoc. Templates that should have them include them at template time.

Post-processor strength is scaled by `max(senderVoice.confidence, POST_PROC_STRENGTH_FLOOR)`. At full confidence (50+ msgs), full projection. At 0 messages, only 30% — we don't impose habits we haven't observed.

### 6.2 Structural rule changes

**Reversal — drop the blanket `/!{1}/`, replace with `/!{3,}/`:**
The current rule bans **any** exclamation mark. Per MARKET_SCAN §1.11, zero exclamation marks is itself an AI tell ("AI overcorrection"). Replace with `/!{3,}/` (ban only triple-or-more exclamation runs). Casual and quick tone templates may now use a single `!`.

**Weakening — tighten `'consider '`:**
The current `'consider '` substring nukes "consider it done," "consider yourself warned." Tighten to `/\bconsider\s+(asking|trying|reaching)/i`.

### 6.3 Compound "too polished" flag

A new structural check fires when **all four** are true:
- `noTypos`: `typoRateInLine == 0`
- `noFragments`: every sentence-piece has ≥3 words
- `noLowercaseStart`: the line starts with an uppercase letter
- `length > 80`

When all four hit, lint returns `{ ok: false, reason: 'too_polished' }`. The composer retries with a lower-register template. This catches the §1.7 / §1.9 / §1.10 cluster ("uncanny valley of niceness") that no single phrase ban would catch.

### 6.4 Post-2025 tells — "AI signature" check

A second compound check:
```
isAISignature = (line.includes('!') === false)
             && (typoCount(line) === 0)
             && (sentenceCount(line) >= 2)
             && (every sentence ends with terminal punctuation)
```
When `isAISignature`, lint returns `{ ok: false, reason: 'ai_signature' }`. **Because:** MARKET_SCAN §1.7 + §1.10 + §1.11 in combination is the single most reliable post-2025 detector.

### 6.5 Tricolon detector

`/\b\w+,\s+\w+,\s+and\s+\w+/` — "bold, thoughtful, and playful" pattern (MARKET_SCAN §1.14). Returns `{ ok: false, reason: 'tricolon' }`. **Because:** humans don't list three adjectives in chat.

### 6.6 Linter API stays backward-compatible

`lintMove(line: string)` keeps the same signature. The only addition is `reason` codes expand from `forbidden:*` and `too_long` to also include `too_polished`, `ai_signature`, `tricolon`. Existing tests must still pass; new tests assert each new code fires on a canned positive example and not on a clean negative.

## 7. Voice Fingerprint Reveal UI

### 7.1 Purpose — Tier-A unique feature

After a sender has 50+ messages (the `senderVoice` full-confidence threshold), surface a one-time **"Your voice fingerprint"** screen. The user sees themselves — not their match. This is the antithesis of an AI-generated message and reinforces the human-feel brand promise.

**No competitor ships this.** Hinge has profile insights, Tinder has Smart Photos, Bumble has Opening Moves analytics for paid users — none surface the user's own chat-style fingerprint as a shareable artifact.

### 7.2 Trigger

When `senderVoice.messageCount >= 50` AND `User.flags.voiceFingerprintShown !== true` AND the user is on the Move surface, surface a one-time bottom-sheet on Move open. After it's shown once, it's reachable from Profile → "Your voice fingerprint."

### 7.3 Content (all derived from `SenderVoiceVector`)

```
You write like a [archetype label].

Your median message is [lenChars.median] characters.
You use emoji [round(emojiRate * 100)]% of the time.
Your top emoji: [topEmojis[0]]
Lowercase 'i' rate: [round(lowercaseIRate * 100)]%

Your top 3 phrases:
- [phrase 1]
- [phrase 2]
- [phrase 3]

Your archetype: [archetype]
[one-line description of archetype: wordsmith / voice_first / visual / fast_replier]
```

The "top 3 phrases" are extracted server-side via a separate pure function `extractTopPhrases(messages, k=3)`:
1. Tokenize each message into bigrams and trigrams.
2. Frequency-count across the 50 messages.
3. Filter out stopwords-only n-grams and dictionary-only n-grams below `idf > 1.5`.
4. Return top 3 by `frequency * idf`, with `frequency >= 3`.

If fewer than 3 phrases meet the bar, show 2 or 1; never pad with stopwords.

### 7.4 Viral mechanic

A **"Share to Instagram Story"** button generates a 1080×1920 PNG (server-rendered via the existing `services/web` static-asset path) containing:
- Branded background (Miamo gradient, no PII).
- The user's archetype label in large type.
- 3-4 stat lines (length, emoji rate, top emoji, lowercase-i %).
- No actual message content.
- "Get your fingerprint at miamo.app" footer.

The PNG is generated on demand, signed-URL'd with a 5-min expiry, served from `/api/v1/users/me/voice-fingerprint.png`. Never cached server-side beyond the 5-min URL — privacy.

### 7.5 Privacy guarantees

- The fingerprint is computed on data the user already wrote and already owns.
- Only the user sees their own fingerprint. **Never displayed in another user's profile, never shared in match metadata, never indexed.**
- The fingerprint PNG includes no message content — only derived stats.
- An explicit "Delete my voice fingerprint" button in Settings → Privacy clears the `senderVoiceV8` cache and the `User.flags.voiceFingerprintShown` flag.
- DPDP-safe: this is processing-by-consent (the user opened the screen).

### 7.6 What it isn't

- Not a leaderboard. No comparisons across users.
- Not an evaluation. No "your voice is 7/10 attractive."
- Not predictive. No "users with your voice match X% more often."
- These are explicit anti-patterns — any of them would convert a delight feature into a status feature, which is the wrong frame for serious dating.

## 8. AI Cringe Self-Test (Adversarial)

### 8.1 Purpose

Continuously verify that composer output does *not* trip a deterministic AI-detector heuristic. This is an internal red-team, not an externally-shipped check.

### 8.2 The detector

A pure function `aiDetectorScore(line: string): number` returning 0..1 (higher = more AI-like). The detector is a weighted sum over:

```ts
const DETECTOR_WEIGHTS = {
  hasEmDash:           0.25,  // because: highest-precision single tell (MARKET_SCAN §1.1)
  zeroExclamations:    0.10,  // because: §1.11; only weighted when line is ≥2 sentences
  zeroTypos:           0.05,  // because: §1.9; weak alone, strong in combination
  perfectCaps:         0.10,  // because: §1.7
  hasSemicolon:        0.15,  // because: semicolons in chat are an LLM signature
  hasTricolon:         0.15,  // because: §1.14
  hasHedgePhrase:      0.10,  // because: §1.12; checks against the §6 cluster
  hasOverPreciseWord:  0.10,  // because: §1.4; "chromatic", "negative space", "meditative"
};
```

**(F8.1) AI detector score**
```
aiScore = sum(weight * (1 if condition else 0)) for all rules
       (capped at 1.0)
isFlagged = aiScore > 0.35
```

### 8.3 Test

Run the composer over a fixed test fixture of 100 pair contexts (varied senderVoice, receiverResonance, hookCandidate, languageFamily, viewerIntent). For each pair, compute 5 suggestions = 500 lines. Compute `aiScore` per line. **Target: fewer than 25 of 500 lines (5%) flagged.**

This test runs in CI on every PR touching `algo/v8/moveV2/`. A regression above 5% blocks merge.

## 14. Risks & Open Questions

1. **Code-mix detection on under-50-message users.** The 200-trigram profile may misclassify users with fewer than 20 outbound messages. Mitigated by the 0.6 confidence floor + English fallback.
2. **Festival hook double-emit.** If Section B emits 2 festivals on the same day (Diwali + a regional festival), the hook library must dedupe. Open: which gets priority? Proposal: regional > national; ranks by user's `Profile.city` region match.
3. **Voice-fingerprint screen on a returning user with new behavior.** A user who churned and came back may have a stale fingerprint. Open: do we recompute when the user explicitly opens it, even if cached? Proposal: yes, force recompute on screen open.
4. **Tanglish/Banglish coverage.** v3.6 ships 80 templates but the realistic linguistic coverage is shallow. We may discover bad fits at 0.1 ramp. Mitigated by `ALGO_V8_MOVE_V2_LANG_FAMILY_FORCE=en` kill switch.
5. **Post-processor over-correction.** Aggressive lowercase / contraction projection might bend a formal user's voice into a casual one. Mitigated by `POST_PROC_STRENGTH_FLOOR` and confidence scaling.
6. **Linter false-positive on legitimate words.** "Genuinely" appears in benign user speech; the new ban might over-fire. Mitigated by AB-testing the linter additions against held-out real user messages (per MARKET_SCAN §6 recommendation).
7. **`FirstMoveOutcome` data sparsity.** Many users won't have 3 successful replies in their last 10. We rely on the archetype cold-start path heavily early on. Open: do we backfill `FirstMoveOutcome` from historical `Message` data? Proposal: yes, one-time backfill job, gated by consent.
8. **DTM-track Move v2.** This design covers Discover-track Move v2. DTM uses the same modules but layers in `dtmCompatibility.ts` topic agreement as a higher-prior hook category and applies the §10.5 safeguards (no Move when `moodGuess < 0.4`).
9. **Tracking pipeline pressure.** Five suggestions per match per day at scale could increase ingest volume by ~3× on Move-related events. Mitigated by aggregation at the worker layer.
10. **Voice-fingerprint share PNG generation cost.** Generating a PNG on demand at peak share-burst (e.g. post-launch viral moment) could overwhelm the web tier. Mitigated by signed-URL caching at the CDN layer with 5-min TTL.

## 15. Self-critique (per prompt §8.3)

1. **Biggest assumption.** That ~40% accept rate is reachable with a pure-template approach. v3 baseline is ~28%; the jump assumes voice-projection and hook-specificity together close most of the gap.
2. **Competitor copy-time.** A competitor can copy the templates in 2 weeks, but not the `FirstMoveOutcome` ledger nor the §7 voice fingerprint. The moat is the cumulative receiver-resonance data, plus the fingerprint as a viral primitive.
3. **Ramp 1.0 failure mode.** Most likely: a small cohort of high-volume users on Hinglish gets over-fit by the post-processor and complains in app reviews. Detection is via the 1% live AI-detector sample.
4. **Complexity not asked for.** The voice fingerprint UI was not in the literal §10.4 spec. It's load-bearing for the brand promise and for virality, but it doubles the surface area of the section. The 50%-of-value version (§15.12) ships without it.
5. **Signal tracked but unused.** `topEmojis` is computed but only used by the fingerprint UI. Audit at the 0.5 ramp; if fingerprint share rate < 5%, consider dropping.
6. **AB-good, retention-bad metric.** Accept rate is gameable by surfacing the fastest-to-tap suggestion. Mitigated by reporting reply rate as the primary outcome, not accept rate.
7. **Existing pattern reinvented.** `moveProfile.ts` already classifies archetype. We layer `senderVoice` on top; we do not re-classify.
8. **Conflict with learner ramps.** Section A's `intentRightNow.ts` ramp is 0.1 currently. If Move v2 ramps faster, we depend on a stable signal that may itself be in flux. Mitigated by Move v2 ramp 0 lasting 7 days until intent is at 0.3+.
9. **Worst-case adversarial.** A bot account sends 50 generic messages to get its `senderVoice` cached, then uses Move v2 to compose on every match. The composer's diversity rule + linter catches generic, but resonance is hollow. Mitigated by gating Move v2 behind reciprocity-weighted match score from Section D.
10. **`INTENT_INFERENCE_ENABLED=0` resilience.** Section A turned off → `viewerIntent` is null → `intentFit` weight (0.10) drops; the other three sub-scores renormalise. The composer still runs.
11. **DTM worse for Discover better?** No — DTM gets the same modules plus extra safeguards. If anything, the DTM topic hook category gives DTM a hook source Discover doesn't have.
12. **50%-of-value version.** Ship Section 1 (senderVoice) + Section 5 (composer, English-only, no code-mix) + Section 6 (linter additions) only. Skip §2, §3, §4, §7 for v3.6.0. Defer to v3.6.1. The principal-engineer instinct says ship the full version because §2-§4 are the *whole point* of the redesign — but the smaller version is the safe fallback if Phase 3 timeline slips.

## D.1 DTM safeguard layer

### D.1.1 Right-now-mood gate

DTM topics are not interchangeable on a bad day. The 16-topic canonical order includes `intimacy` (index 3), `finance` (5), `conflict` (6) — topics with empirically higher abandon rates when Section A reports `moodGuess < 0.4` or Section B reports `windowShopping=true` for the recent session window.

**The gate is a soft skip, not a fail.** A skipped topic stays in the user's eligibility pool and re-surfaces at the next session that clears the gate. The batch builder backfills with light topics so the user never sees a degraded experience.

Let `M = moodGuess ∈ [0,1]` (neutral 0.5 when missing), `WS_2 = true` iff the last 2 completed sessions both had `windowShopping=true`, `H = { intimacy, conflict, finance }`.

```
(1)   gated(T)  =  T ∈ H  ∧  ( M < 0.4  ∨  WS_2 )
```

- `0.4` mood threshold — **because:** Section A's polarity-classifier negative-confidence band starts at 0.35 (one-sigma below neutral 0.5 in pilot logs); 0.4 gives a 5-point cushion so a flat-but-not-negative session doesn't strip heavy topics.
- `WS_2`, not `WS_1` — **because:** a single window-shopping session is normal mid-week behavior; two consecutive is the published threshold above which message-reply rate drops >15% (`MARKET_SCAN.md` §4 session-based recommenders).

### D.1.2 Defer-to-next-session semantics

Skipped topics are recorded in `FeatureSnapshot.raw.dtmMoodMask`:

```json
{ "deferredTopics": ["intimacy","conflict","finance"], "computedAt": 1719252000000, "ttlMs": 86400000 }
```

The next DTM batch after TTL expiry (24 h) re-evaluates `gated(T)` from fresh inputs. If mood recovers, deferred topics re-enter naturally — no special deferred-bucket code path. The existing v7 saturation/skip/abandon penalties still apply on top of the mask: a topic the user has actively skipped 3 times in 3 days remains penalised whether the mood gate triggered or not.

### D.1.3 `dtmFeedV7.ts` change — accept `topicMask` param

Pure module stays pure. The caller (DTM batch route in `services/content/src/server.ts`) computes the mask and passes it in.


## D.3 Family Brief — Tier A unique feature (India)

### D.3.1 Concept

Signed-in user with a populated `MatrimonialProfile` taps **Share with family** on profile-detail. They pick a format (PDF / image / plain-text). Server renders a traditional Indian bio-data card and creates a `FamilyBriefShare` row with a token. The user gets a short URL plus the artifact; they paste either into WhatsApp. The parent opens the URL, sees a watermarked mobile-friendly bio-data, and (privacy-respecting) `viewCount` increments. URL is dead at `expiresAt`.

### D.3.2 Format options

| Format | Use case | Size | Source-of-truth |
|---|---|---|---|
| **PDF** | Email / print | A4, ~150 KB | Server-rendered with `puppeteer-core` from HTML template |
| **Image (1080×1920 PNG)** | WhatsApp Story | <500 KB | Same HTML at story aspect via `--screenshot` |
| **Plain text** | Copy into chat | ~2 KB | Separate templated string |

PDF and image share one HTML template (`services/content/templates/family-brief-v1.html`) — they differ only in viewport. **Because:** maintaining two render paths invites layout drift; one template guarantees parity. Plain text is rendered from a separate stub to avoid HTML-stripping artefacts.

### D.3.3 Layout — traditional Indian bio-data

Order matches the cultural convention from the India scan:

1. Title: `Bio Data` + full name in large type
2. Photograph (primary, square-cropped)
3. Personal: DOB, time of birth, place of birth, height
4. Religion / star (nakshatra) / raasi / gotra / manglik
5. Education + College
6. Profession + Company + Annual income band
7. Family: father's & mother's names + occupations, siblings (count + married)
8. Partner expectations (free-text from `partnerExpectation`, capped 600 chars)
9. Watermark: `Created on Miamo • <short-url> • Expires <date>`

Caste / sub-caste appear **only if the user filled them in**. The Family Brief is the one surface where user-supplied caste renders — because the parent-share UX is meaningless to caste-conscious families if absent. **Consistent with the founder decision:** caste is user-owned data, never input to any algorithm; this is a literal export of the user's field. `FEATURE_FAMILY_BRIEF_HIDE_CASTE=1` suppresses caste globally if a regulator requires; default `0`.

### D.3.4 Privacy & TTL

- **TTL: 7 days.** **Because:** parent-side decision cadence median is 3–10 days (`MARKET_SCAN.md` §5 §3). Re-share is one tap.
- **Token:** 22-char base64url via `crypto.randomBytes(16).toString('base64url')`. **Because:** matches existing `hashUid` width — collision probability ~10⁻²⁰ at 1 M outstanding tokens.
- **Opt-in tracking:** at share time, the user toggles "show me how many parents viewed this (anonymous count)." If on, viewCount increments on each fetch, debounced 60 s per IP, IP discarded. If off, the column stays 0.
- **Optional label:** `sharedTo` is a free-text ≤40-char label ("Mom", "Riya's parents") for disambiguation.
- **No referer leak:** share page renders server-side with one fetch, inline CSS, zero third-party scripts. **Because:** DPDP §1 requires lawful-basis disclosure for any cross-site recipient.

### D.3.5 Routes

```
POST /api/v1/dtm/family-brief/generate

  auth: required (JWT)
```

## D.4 Mood-matched DTM topic queue — Tier B

### D.4.1 Module — `algo/v8/dtmTopicMask.ts` (pure)

```ts
// services/shared/src/algo/v8/dtmTopicMask.ts
import { DTM_TOPIC_KEYS, type DtmTopicKey } from '../dtmTopics';
import type { DtmColdStartStage } from '../dtmColdStart';

export type SessionHistory = { windowShoppingLast2: boolean };

const HEAVY_TOPICS: readonly DtmTopicKey[] =
  ['intimacy', 'conflict', 'finance'] as const;
// because: highest abandon rate on low-mood sessions (Section A pilot Mar–May 2026).

const LIGHT_TOPICS: readonly DtmTopicKey[] =
  ['values', 'lifestyle', 'communication', 'leisure'] as const;
// because: lowest abandon rate at coverage 'sparse'; highest completion on cold-start.

const MOOD_THRESHOLD = 0.4;
// because: Section A polarity negative band starts at 0.35; 5-point cushion.

export function computeTopicMask(
  viewerMood: number,
  history: SessionHistory,
  dtmCoverage: DtmColdStartStage,
): DtmTopicKey[] {
  const mask = new Set<DtmTopicKey>();
  if (viewerMood < MOOD_THRESHOLD || history.windowShoppingLast2) {
    for (const t of HEAVY_TOPICS) mask.add(t);
  }
  if (dtmCoverage === 'empty' || dtmCoverage === 'sparse') {
    for (const t of DTM_TOPIC_KEYS) {
      if (!LIGHT_TOPICS.includes(t)) mask.add(t);
    }
  }
  // sufficient+ → no additional mask
  return Array.from(mask);
}
```

```
(3)  mood < 0.4 ∨ windowShoppingLast2     →  mask += {intimacy, conflict, finance}
(4)  coverage ∈ {empty, sparse}           →  mask += (DTM_TOPIC_KEYS \ LIGHT_TOPICS)
(5)  coverage ∈ {sufficient, full}        →  no additional mask
```

The mask is **monotone** — stronger conditions accumulate. A sparse-stage user with mood < 0.4 sees only light topics, which is the correct strict behaviour for a low-mood new user.

### D.4.2 Wiring and tracking

Computed once per DTM-batch request in `services/content/src/server.ts` (D.1.4). Each excluded topic emits one `dtm.topic_masked` event with reasons (`'mood' | 'windowShopping' | 'coverage_sparse'`). Telemetry consumes for the KPI in D.12.

### D.4.3 Acceptance test — "never asks intimacy at 11 pm on a 0.3-mood night"

```ts
test('11pm low-mood night strips heavy topics', () => {
  const mask = computeTopicMask(0.3, { windowShoppingLast2: false }, 'sufficient');
  expect(mask).toContain('intimacy');
  expect(mask).toContain('conflict');
  expect(mask).toContain('finance');
  expect(mask).not.toContain('values');
  expect(mask).not.toContain('lifestyle');

```

Plus: a sparse-coverage test strips everything except light topics; a property test verifies the result is a subset of `DTM_TOPIC_KEYS` with no duplicates.

## D.5 Anti-ghost economy — Tier B

### D.5.1 Mechanism

Sender deposits 1 Spotlight minute when **starting** a new chat (zero prior messages from sender to receiver). The deposit is held in escrow on `SpotlightLedger`. If receiver replies within 72 h, the minute is **credited to the receiver**. If neither replies within 72 h, the minute **burns**. If sender ghosts (no second message after match within 7 d), their next chat-deposit costs **2 minutes**; reset after one successful conversation (≥3 reply rounds).

| Event | Sender | Receiver | Pool |
|---|---|---|---|
| Sender starts new chat | −1 (or −2 if penalised) | 0 | +1 escrow |
| Receiver replies within 72 h | 0 | +1 | −1 escrow |
| Neither replies 72 h | 0 | 0 | −1 burn |
| Sender no-second-msg after match | 0 (penalty flag set) | 0 | 0 |
| First successful conv. (≥3 rounds) | 0 (penalty cleared) | 0 | 0 |

### D.5.2 Math

`B_send` = sender's lifetime balance. `r_reply` = receiver's reply probability within 72 h. `D = 1` or `2` (penalty).

```
(6)  E[cost_sender]    =  D · (1 - r_reply)    (deposit not refunded to sender; receiver gets it)
(7)  E[gain_receiver]  =  D · r_reply          (rewarded for replying)
```

The sender's deposit is a **signal cost**: a serious sender pays it gladly; a spammer accumulates losses. Receiver doesn't need to know to be incentivized — surfacing the bonus to them is a separate UX choice (recommended: show only after reply, as "+1 minute for replying").

### D.5.3 Constants

- `DEPOSIT_BASE = 1` minute. **Because:** matches existing `daily_login` Spotlight grant — calibrated noticeable, not punitive for serious users.
- `DEPOSIT_PENALTY = 2` minutes. **Because:** doubling after one ghost is the minimum visible escalation that survives confirmation-bias.
- `REPLY_WINDOW_HOURS = 72`. **Because:** Hinge's median time-to-first-reply is 18 h; 72 h covers >95% of organic distribution.
- `GHOST_PENALTY_AFTER_MATCH_HOURS = 168` (7 d). **Because:** founder definition; 7 d excludes legitimate slow-replier behaviour.
- `MAX_CHAT_DEPOSITS_PER_DAY = 10` per user. **Because:** anti-grief; 10 min cap is recoverable in a single login streak.
- `PREMIUM_DEPOSIT_MULTIPLIER = 0.5`. **Because:** founder's "priority not free pass." Implemented as a daily reconciliation row (`chat_deposit_premium_refund`) refunding 50% of the day's `chat_deposit` minutes, rounded down — preserves the ledger's "every flow is one row" property and keeps deposits integer-valued.

### D.5.4 SpotlightLedger reasons (no schema change)

`SpotlightLedger.reason` is `String` already (line 717). Four new values:

- `chat_deposit` — `delta = -1` (or `-2`), `refId = chatId`, `meta = { receiverId, depositedAt, escrowToken }`
- `chat_reply_bonus` — `delta = +1`, `refId = chatId`, `meta = { depositerId, replyMs }`
- `chat_ghost_burn` — `delta = -1` charged to system row, `refId = chatId`, `meta = { senderId, burnedAt }` (sender is not double-debited; the original deposit row stands)
- `chat_deposit_premium_refund` — `delta = +1` per refunded minute, `refId = null`, `meta = { dateUtc }`

### D.5.5 Worker — `chatDepositReconciler.ts`

Runs every 5 minutes. For each `chat_deposit` row older than 72 h with no matching reply:

1. Check `Message` table: receiver message since deposit?
   - Yes → write `chat_reply_bonus` (idempotent on `(refId, reason)`).
   - No → write `chat_ghost_burn`.
2. Mark deposit reconciled via `meta.reconciledAt`.

Idempotent — re-running over the same window is safe (app-layer uniqueness check before insert). Daily 00:30 UTC sweep computes premium refunds.

### D.5.7 Ghost-detection helper


## D.7 DTM cold-start improvements

Stage thresholds untouched. Improvements are **about which topics to ask in what order**.

### D.7.1 Topic prioritisation by coverage stage

The D.4 mood-mask enforces light-only at sparse. Within the light set, order is:

```
(8)   Stage 'empty'/'sparse' →  [values, lifestyle, communication, leisure]
(9)   Stage 'sufficient'      →  unlock [growth, family, faith, ambition, autonomy, social, health, parenting, future]
(10)  Stage 'sufficient' + mood ≥ 0.4 + !WS_2 → unlock [intimacy, conflict, finance]
```

The (8) order maximises first-7 diversity: values (world-view) → lifestyle (daily habit) → communication (relational style) → leisure (hobby) — one topic from each of four semantic clusters.

### D.7.2 First-7-topics rule

Within the first 7 topics, ≥1 from each cluster:

- **values:** `values`, `faith`
- **lifestyle:** `lifestyle`, `leisure`, `health`
- **social:** `communication`, `social`, `family`
- **future:** `future`, `growth`, `ambition`, `parenting`

Enforced via a `categoryFloor: Record<ClusterId, number>` advisory to `buildDtmFeed`. Pure module re-orders the batch to meet the floor. **Because:** users who only see one-cluster questions report feeling "boxed-in" (`MARKET_SCAN.md` §5 §3). If the mood-mask removes all candidates from a cluster, the rule degrades to "≥1 from each cluster *that has at least one available topic*."

### D.7.3 Culture-aware question text

Each DTM topic's question text is rendered through a templating layer that reads the user's `culture` field (`Settings.culture` or a deterministic name-heuristic fallback).

| Culture | Rendered for `family` topic |
|---|---|
| `indian-north` | "How often do you expect to visit family on weekends after marriage?" |
| `indian-south` | "How important is your family's blessing for major life decisions?" |
| `diaspora-indian` | "How do you balance independence and family expectations?" |
| `secular-global` (fallback) | "What role does family play in your day-to-day life?" |

Phrasings live in `services/content/src/dtm-question-text/{culture}.json`. The renderer is pure. **Scope is strictly text rendering** — DTM vectors and topic semantics are unchanged. No vector rebuild.

### E.1.6 Worked example — design-around for §10.1 if Match Group reads on it

The most likely conflict: Match Group's US 2025/0012345-class "session intent classification." If the granted claims cover "automatically classifying a dating-app session into one of N intent classes from short-window behavioural features," we have three credible design-around paths, ranked by founder preference:

**Path 1 — opt-in toggle (cheapest, ships fastest).**
- Add a Settings toggle `sessionIntentEnabled` (default OFF).
- Engineering change: `intentInference.ts` only runs when `Settings.moodInferenceEnabled = true` *AND* `Settings.sessionIntentEnabled = true`.
- Claim differentiation: Match Group's claim almost certainly recites "automatically classifying," which an opt-in user-toggled inference is not (the user is the classifier; the system is the prompted helper).
- Cost: ~15% of users will turn it on at v3.6.0 launch; we expect 40-60% adoption by month 3 once Why-am-I-seeing-this (E.3) demonstrates value. Reduces effective ramp by ~50% in week 1.

**Path 2 — declarative intent picker (no inference at all).**
- Replace 7-class intent inference with a 4-button picker shown on session start: `Casual browse / Focused search / Replying to threads / Just looking at the app`.
- Inference module disabled entirely until v3.7.
- Claim differentiation: no automated classification → no read.
- Cost: ~20% of users will pick, the rest will not. Drops Section A's main feature from "live re-rank" to "manual mode."

**Path 3 — different feature space (most expensive).**
- Re-derive intent from a feature space *not* in Match Group's claims — e.g. infer from *physical environment* signals (timezone × device-orientation × ambient light if exposed by browser, NOT facial emotion). Stay outside the claim's "session behavioural features" language.
- Cost: meaningful re-design of Section A. ~3 weeks delay to v3.6.0.

**Recommendation:** ship Path 1 by default. Section A's `intentRightNow.ts` is already gated on `moodInferenceEnabled === true`; the additional `sessionIntentEnabled` sub-toggle is one extra field on `Settings` and one extra branch in the worker.

### E.1.7 What happens if clearance fails outright

If two or more Match Group issued patents read on Sections A/B/C and design-around paths add >3 weeks to the calendar:

- v3.6.0 RC ships **Sections C + D only** (Move v2 + DTM safeguards + Family Brief + Voice-note + Anti-ghost). Sections A and B are deferred to v3.6.1 or v3.7.
- The Move v2 composer in Section C is, by design, low patent risk — it operates on the sender's own profile and the receiver's *self-published* profile content, not on inferred receiver mood.
- Section D's DTM mood gate is local-to-user (no cross-user inference) and is on the founder's IP, not Match Group's.

This is a non-trivial outcome and the design-doc must be explicit that the founder accepts it as a possible path.

## E.2 Privacy & consent

Market scan §6 reviewed DPDP 2023, GDPR (specifically Articles 9 and 22), CCPA, ATT, and ITP. The synthesis: **mood inference, session intent, and depth-of-engagement signals are likely special-category data under GDPR Article 9 and analogous DPDP categories.** Miamo's privacy posture must be granular, default-conservative, and per-inference revocable.

### E.2.1 Five new `Settings` toggles

Add to `services/users/prisma/schema.prisma`:

```prisma
model Settings {
  // … existing fields …
  moodInferenceEnabled        Boolean @default(false)
  // because: mood is special-category data under GDPR Article 9;
  // default OFF until user explicitly opts in — required for any §10.1 work
  behavioralRankingEnabled    Boolean @default(true)
  // because: the learner (algo/learner.ts) is the heart of Miamo's product —
  // turning it off degrades to "static rank by recency"; default ON but
  // surfaced in Settings so users can opt out
  crossUserInferenceEnabled   Boolean @default(true)
  // because: pair-compat math (algo/pairCompat.ts) reads other users' profiles
  // to compute compatibility; opt-out means the user's signals do NOT influence
  // other users' queues (their data is not used to train cross-user models)
  algorithmicTransparency     Boolean @default(true)
  // because: "why am I seeing this" tooltips (E.3) are a feature, not a privacy
  // toggle; default ON because we want everyone to see them
  sessionIntentEnabled        Boolean @default(false)
  // because: patent design-around (E.1.6 Path 1); only run intentInference.ts
  // when this is true, to avoid Match Group claim language
}
```

### E.2.3 Per-inference opt-out paths (granular)

The five toggles map to specific code paths. The mapping is enforced at call sites — *no algo module reads PII without consulting the toggle*. The existing `withConsent(userId, scope, resolver, read, fallback)` helper (used throughout `services/shared/src/algo/`) already supports this; we add three new scopes:

| Scope name | Toggle | Modules that gate on it |
|---|---|---|
| `mood-inference` | `moodInferenceEnabled` | A.intentRightNow, D.dtmTopicMask, all of `worker/intentInference.ts` |
| `behavioral-ranking` | `behavioralRankingEnabled` | learner.ts, every algo that reads `MoveDecisionEvent` or `MatchOutcome` |
| `cross-user-inference` | `crossUserInferenceEnabled` | pairCompat.ts, exposureLedger.ts (donor side), C.receiverResonance |
| `algorithmic-transparency` | `algorithmicTransparency` | explain.ts UI surface (not a data scope) |
| `session-intent` | `sessionIntentEnabled` | intentInference.ts specifically (sub-toggle of `mood-inference`) |

When a toggle is off:
- The algo module returns its **neutral fallback** (defined per Section A.7, B.7.3, C.1.8). It never crashes; it never reads the gated data.
- `intentInference.ts` worker **skips** the user entirely. No `FeatureSnapshot.raw.intentRightNow` row is written.
- `exposureLedger.ts` continues to track the user *as recipient* (they still receive impressions) but does not record their signals *as donor* (their actions do not flow into others' fairness math).

### E.2.8 DPIA template

## E.4 AB test harness

### E.4.1 Allocation function

Add `services/shared/src/abTest.ts`:

```typescript
import { createHash } from 'crypto';

/**
 * Stable per-user bucket assignment for an experiment.
 * Returns an integer in [0, 1000).
 */
export function bucketFor(uidHash: string, experimentName: string): number {
  const h = createHash('sha256')
    .update(uidHash)
    .update(':')
    .update(experimentName)
    .digest();
  // because: SHA256 over (uidHash + experimentName) gives a stable, independent
  // bucket per (user × experiment); collision probability between users is
  // 1/1000 per pair, well within tolerance
  const top4 = h.readUInt32BE(0);
  return top4 % 1000;
  // because: 1000 buckets gives us 50/50/900 split granularity without
  // overcomplicating the dashboard
}

export type AbBucketRole = 'control' | 'holdout' | 'treatment-eligible';

export function bucketRole(bucket: number): AbBucketRole {
  if (bucket < 50)  return 'control';
  // because: 50 buckets (5%) is the minimum we can run while still detecting
  // 5% relative effects with α=0.05, β=0.20 at ~50k users per arm
  if (bucket < 100) return 'holdout';
  // because: 50 buckets (5%) reserved for never-see-treatment validity tracking
  return 'treatment-eligible';
  // because: remaining 900 buckets (90%) split by ramp percentage —
  // at ramp 0.1, only buckets 100–189 see treatment; at 0.3, 100–369; at 1.0, all 900
}

export function inTreatment(
  bucket: number,
  ramp: number,  // 0..1
): boolean {
  const role = bucketRole(bucket);
  if (role !== 'treatment-eligible') return false;
  // because: control and holdout never see treatment, regardless of ramp
  const eligibleBuckets = 900;
  const treatmentBuckets = Math.floor(eligibleBuckets * ramp);
  // because: linear ramp across the eligible 900-bucket band
  return (bucket - 100) < treatmentBuckets;
}
```

### E.4.2 Bucket semantics

- **Control (50 buckets, 5%):** never see treatment for *this* experiment. Resets per experiment — a user might be control on Discover and treatment on Move.
- **Holdout (50 buckets, 5%):** **never** see treatment for *this* experiment, **forever**. This is the long-term validity track. Holdout cohort metrics are reported alongside control for the lifetime of the experiment (6+ months) to detect treatment effects that compound over time.
- **Treatment-eligible (900 buckets, 90%):** sub-divided by ramp percentage. At ramp = 0.1, 10% of eligible buckets (90 of 900) see treatment, so the population total treatment is 9%.

The 5/5/90 split is asymmetric on purpose: in a serious-dating app, *long-term* metrics (D30 retention, second-date confirmation) matter more than short-term lift, and a 5% permanent holdout is what lets us read the long-term curve cleanly.

### E.4.3 Power calculation

Target: detect a 5% relative effect on the primary KPI (sym-match → first-message rate, baseline ~25%) with α=0.05, β=0.20.

Standard two-proportion z-test power formula:

```
n_per_arm = 2 × (z_{α/2} + z_β)² × p(1-p) / (Δp)²
         = 2 × (1.96 + 0.84)² × 0.25 × 0.75 / (0.0125)²
         = 2 × 7.84 × 0.1875 / 0.000156
         ≈ 18,800 users per arm for a single-comparison test
```

At three simultaneous comparisons (control vs ramp 0.1, vs ramp 0.3, vs ramp 1.0) with Bonferroni correction (α = 0.05/3 = 0.017, z = 2.39):

```
n_per_arm ≈ 30,000 users per arm
```

Adding 50% margin for noise and weekday effects:

```
n_per_arm ≈ 50,000 users
```

This sets the **minimum population** required to run an AB ramp on a primary KPI. Miamo currently has ~120k MAU (per knowledge base); the math works if and only if we have at least 50k active users in the comparison cohort within the 14-day window. We do.

For guardrail KPIs (D1 retention, session length, crash rate), the effect we must detect is *smaller* (1% absolute drop); n_per_arm scales to ~150k. We do *not* have 150k per arm, so guardrail detection is **at lower power** than primary detection. This is documented in E.8 risk #11 and again in the self-critique (E.9 question 1).

### E.4.4 Stop rule

Already in prompt §9.4; restated for completeness:

- Any **guardrail KPI** dropping >1% (absolute for D1/D7 retention; relative for session length and crash rate) versus control over the last 48h of the ramp window → **automatic rollback to the previous ramp step**.
- Auto-rollback mechanism: a worker `abGuardrailWatcher.ts` reads the KPI dashboard every 6 hours; if a guardrail breach holds across two consecutive reads (i.e. is not a transient), it flips the relevant `ALGO_V8_*_ENABLED` flag back to the previous ramp and posts a Sev2 issue to the on-call channel.
- The rollback is reversible after RCA; the *automatic* part is the safety net, not the final decision.

Every algo module that has a v8 path checks `inTreatment(bucket, ramp)` near the entry. Pattern:

### E.4.7 Adversarial AB allocation — risk addressed

Risk #10 in E.8: someone deliberately gaming experiment buckets by creating multiple accounts to land in treatment.

Defense:
- The bucket function uses `uidHash`, which is itself HMAC over `userId` with a server-side secret. An attacker cannot predict their bucket without knowing the secret.
- The `experimentName` is server-side; clients don't pick it.
- Combined: a sybil attacker would need O(1000) accounts to land one in a specific bucket, which is detectable via the rage-likes downweighter in Section B.4.1.

```typescript
// KPI #1 — Sym-match → first-message rate (48h)
export async function kpiSymMatchToFirstMessage48h(window: TimeWindow): Promise<number> {
  const result = await prisma.$queryRaw<{ rate: number }[]>`
    WITH matched AS (
      SELECT m.id AS match_id, m."chatId" AS chat_id, m."createdAt" AS t0
      FROM "Match" m
      WHERE m."createdAt" >= ${window.start}
        AND m."createdAt" <  ${window.end}
    ),
    messaged AS (
      SELECT DISTINCT mt.match_id
      FROM matched mt
      WHERE EXISTS (
        SELECT 1 FROM "Message" msg
        WHERE msg."chatId" = mt.chat_id
          AND msg."createdAt" BETWEEN mt.t0 AND mt.t0 + INTERVAL '48 hours'
      )
    )
    SELECT (COUNT(*) FILTER (WHERE m.match_id IN (SELECT match_id FROM messaged)))::float
           / NULLIF(COUNT(*), 0) AS rate
    FROM matched m;
  `;
  return result[0]?.rate ?? 0;
}
// because: window-bounded so the daily dashboard reads a fresh 48h cohort each run
```

Then: `npm run prisma:generate` from `services/shared`, restart all three node services. The migration applies once (at the Postgres level); the schema mirror is for the type system only.

### E.6.3 Forward-only

No `DROP`s. If we need to remove a column, we tombstone it (set to NULL) and the next major version (v4.0.0) drops in a separate migration. Reason: rolling back v3.6.0 must leave the schema intact so a v3.5.1 service can still run against the migrated DB.

### E.6.4 Schema-failure on production — risk #11

Postgres has had a history of failed-mid-migration incidents on this project (per project memory). Mitigation:

- Each `CREATE TABLE` is wrapped in `IF NOT EXISTS`.
- Each `ALTER TABLE` is wrapped in a `DO $$` block.
- A pre-migration script reads the current schema and reports drift.
- The migration is run in a single transaction (Postgres supports transactional DDL).
- A staging-environment dry-run is mandatory before the prod run.

### P0 — could kill the launch

**Risk #1: Match Group patent clash on real-time intent (Section A / §10.1).**
- Description: Match Group filings 2024–2026 may read on `intentRightNow.ts`. If granted and asserted, v3.6.0 Section A is unshippable in the US.

**Risk #2: DPDP penalty exposure if consent screens insufficient.**
- Description: India's DPDP allows ₹50Cr per breach. If our consent UI is later judged "not informed consent" (vague checkboxes, dark patterns), every existing user becomes a violation.
- Owner: founder + DPO.
- Mitigation: E.2.4 / E.2.5 consent flows reviewed by Indian counsel before launch; granular per-inference toggles (E.2.3) ensure user retains control post-consent.
- Kill criteria: Indian counsel review returns "not informed consent" verdict; pause launch until UI is revised.

**Risk #3: Family Brief PII leak via shareable URL.**

- Owner: engineer.

**Risk #4: Gini blowing past 0.45 due to fairness math bug.**
- Description: Singh-Joachims rerank (Section B.8) miscalibrates and pushes Gini past 0.45 instead of bringing it down.

## E.9 Self-critique (the §8.3 12-question pass)

Honest answers per prompt §8.3:

**1. What's the biggest assumption I'm making that could be wrong?**
That a 5% permanent holdout is enough statistical power for D7 retention guardrail detection. At our MAU and 1% effect threshold, the math (E.4.3) says we need ~150k per arm; we have ~60k per arm post-holdout. Guardrail detection is *underpowered*. The risk is we ramp something that subtly degrades D7 and don't notice for weeks. **Mitigation:** dashboard runs nightly so even underpowered detection finds the drift within 14 days; the 5% holdout cohort is preserved across all experiments so cumulative effects show up.

**2. If a competitor copied this design in 2 weeks, what's our moat?**
- The Family Brief (Section D.3) — culturally specific, unlikely to be copied by global apps.
- The voice-fingerprint reveal (Section C.7) — Miamo-specific affective surface; competitors would build a different version.
- The "why am I seeing this" in-card explainer (E.3) — none of Tinder/Hinge/Bumble ships this; copying it takes ~6 weeks because `explain.ts` is a real engineering artefact, not a UI choice.
- The Singh-Joachims fairness math (Section B.8) — public paper, copy-able. *Not* a moat. Our moat there is the *operational discipline* (Gini alerts, gender-conditional version) not the math.

**3. What's the failure mode at ramp 1.0 that I haven't tested for?**
Cross-surface interaction. Discover ramp 1.0, DTM ramp 1.0, Move v2 ramp 1.0 *all on the same user* — the compound effect of three new rankers reading from the same exposure ledger may push fairness Gini past 0.45 in a way that's invisible at single-surface ramp testing. **Add to Phase 3 test plan:** a cross-surface stress test in `tests/cross-surface-ramp.test.ts` that simulates 1000 users hitting all three surfaces and measures Gini.

**4. Where am I adding complexity that the founder didn't ask for?**
- The 5% permanent holdout cohort. Founder asked for AB testing; they did not ask for a 6-month validity track. Justification: long-term metrics in serious dating matter more than short-term lift. **Stays.**
- The `/algorithm-transparency` public web route. Founder did not ask for a public transparency page. Justification: Article 22 compliance + competitive moat. **Stays.**
- The gender-conditional Gini. Founder asked for flat Gini ≤0.40. Justification: market scan §1 implication #6 — Indian gender skew makes flat Gini misleading. **Stays, but flagged in E.10 #4 family.**

**5. What signal am I tracking that I won't actually use?**
The "voice-fingerprint reveal" stores `senderVoiceVector` to the user's profile (Section C.7); if no user ever opens the reveal screen, the stored vector is wasted bytes. **Mitigation:** the vector is also read by the composer (C.1), so it's not wasted; the reveal is a side-channel.

**6. What metric will look great in AB but harm long-term retention?**
Move-suggestion accept rate (KPI #6). Users may accept suggestions in week 1 because they're novel; weeks 2–3 see fewer replies because the receiver detects AI signature → unmatch. This is exactly why KPI #7 (suggestion→reply ratio) is also primary, not just accept rate. If #6 climbs while #7 falls, **trust #7**.

**7. Is there an existing Miamo pattern I'm reinventing?**
Yes — the AB harness in E.4. Miamo already has `services/shared/src/featureFlag.ts` with a `getFlag(userId, flagName)` function. The new `bucketFor` is similar. **Decision:** keep them separate. `featureFlag` is boolean on/off per user (engineering rollouts); `abTest` is bucket-allocation per experiment (product experiments). Different semantics. Same project, different tools.

**8. Where does my design conflict with an existing learner ramp?**
The learner ramp in `services/shared/src/algo/learner.ts` adjusts per-user weights every ~30 minutes. The exposure ledger (Section B) earns credit at the moment of a quality action. There's a *timing* conflict: a user who just earned credit may have a stale weight vector that under-uses the credit. **Mitigation:** B.5.6 worker `exposureScheduler.ts` runs every 5 minutes (faster than learner) so credit-spending lag is bounded.

**9. What's the worst-case adversarial user behaviour, and does my design degrade gracefully?**
Rage-likes (1000 likes/min) earn zero credit per B.4.1 → ledger ignores the spam. Sybil deposits per E.8 risk #6 → detected by graph traversal. Spam voice-notes per Section D.6 → length-limited and rate-limited at upload. Profile-scraping bots → `card.impression.50` events with depth=0 don't earn credit either way. **Result:** every adversarial path degrades the attacker, not the platform.

**10. If `INTENT_INFERENCE_ENABLED=0`, does the rest of the system still work?**
Yes. Section A's `intentRightNow.ts` returns a neutral fallback when the flag is off (Section A.7). All downstream consumers (exposure ledger, Move composer, DTM topic mask) handle the neutral case explicitly. The system without intent inference is the v3.5 system plus exposure-ledger plus Move v2 — still a meaningful upgrade.

**11. Have I made DTM worse to make Discover better?**
Looking honestly: Sections A+B introduce ranking changes that primarily benefit Discover (higher impression volume → better learning). DTM benefits from the *same* infrastructure but at slower ramp (E.7), so it's neither helped nor harmed in the short term. Long-term, DTM's mood-gate (Section D.1) is genuinely new and benefits only DTM. **Verdict:** no — DTM is not made worse. But DTM is not made *much* better either in v3.6; v3.7 has more DTM-specific work.

**12. Is there a 50%-of-the-value, 10%-of-the-work version I should ship first?**
Yes: **Section B.1 alone (passed-profile hard filter)** is the 50/10 version. It's a one-line code change, ships in v3.5.1 in Week 1, and addresses the #1 user grievance globally (market scan §2). It is *already* in the Week 1 rollout (E.7). We are not deferring it — we are shipping it ahead of everything else, exactly as the 50/10 principle demands.
