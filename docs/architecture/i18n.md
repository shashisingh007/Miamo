# Internationalisation (i18n)

**Status:** launch scaffold — English + Hindi complete for the top-30 UI strings; Tamil + Bengali have registered catalogues but fall back to English at lookup time until strings are translated.

**Purpose:** ship a working locale switcher for India's four largest launch languages without doing a multi-week rewrite of every component. This document is the how-to for adding coverage.

Cross-references:
- Code: `services/web/src/i18n/{config,useI18n}.ts`, `services/web/src/i18n/{en,hi,ta,bn}.json`
- Wiring: `services/web/src/app/(main)/settings/page.tsx` (language radio group)
- Spec: `FULL_AUDIT_AND_LEARNING_V2_PROMPT.md` §G.13

## 1. Supported locales

| Code | Language | Label | Coverage |
|---|---|---|---|
| `en` | English | English | Source of truth (30 keys) |
| `hi` | Hindi | हिन्दी | Full — 30 keys translated |
| `ta` | Tamil | தமிழ் | Skeleton — falls back to English |
| `bn` | Bengali | বাংলা | Skeleton — falls back to English |

`DEFAULT_LOCALE` is `en`. Adding a locale is a five-line change to `config.ts` plus the new JSON file — see §5.

## 2. Runtime model

There is no top-level React context provider. Each component that needs translations calls the `useI18n()` hook directly:

```tsx
import { useI18n } from '@/i18n/useI18n';

export function ExampleButton() {
  const { t } = useI18n();
  return <button>{t('common.save')}</button>;
}
```

Under the hood:
- On mount the hook reads `localStorage['miamo.locale']`.
- If missing, it detects from `navigator.language` and stores the result.
- `setLocale(l)` writes to localStorage and dispatches a `CustomEvent<Locale>('miamo.locale.changed')` so every mounted `useI18n` subscriber re-renders in sync.
- Server-side rendering always uses `DEFAULT_LOCALE` (`en`). The switcher hydrates on the client and re-renders the UI.

## 3. Lookup + fallback

`translate(locale, key)` in `config.ts` is the pure function every consumer uses. Its fallback chain:

1. Requested locale's catalogue
2. English catalogue (source of truth)
3. The key itself (safety net — never a blank string)

This means a missing translation in `ta.json` never produces a blank UI: users see the English string until the translation ships.

## 4. Translation-workflow rules

1. **English is the source of truth.** Every new key must land in `en.json` first. Only then may it appear in other locale files.
2. **Keys are dot-namespaced.** `nav.discover`, `common.save`, `settings.language`. Match the file structure of the surface you're translating.
3. **Never inline copy that changes with the locale.** If a string is user-visible and will change per language, it goes through `t()`. Hardcoded strings are technical debt; we hunt them with a lint pass in Phase H.
4. **Do not concatenate localised strings.** `t('a') + t('b')` is a smell; each translator needs full-sentence context. Add a new key like `combined.a_and_b` instead.
5. **Numbers, dates, currency:** use `Intl.NumberFormat` / `Intl.DateTimeFormat` with the current locale — never a hardcoded formatter.
6. **Pluralisation:** the current scaffold does not include a plural-forms library. For plural-sensitive strings, ship one key per form (`match.one`, `match.other`) and select at the call-site until we bring in an ICU library.

## 5. Adding a new locale

To add Kannada (`kn`) as a fifth locale:

1. Add `kn` to `SUPPORTED_LOCALES` in `services/web/src/i18n/config.ts`.
2. Add a label to `LOCALE_LABELS`: `kn: 'ಕನ್ನಡ'`.
3. Create `services/web/src/i18n/kn.json`. Populate as many keys as you can; leave the rest blank — the fallback catches them.
4. Import the JSON in `config.ts` and register in `CATALOGUES`.
5. The Settings switcher picks it up automatically because it iterates `locales`.

No other file needs to change.

## 6. Adding a new translation key

1. Add the key to `en.json` with the English source string.
2. Add the same key to every other locale file (best-effort translation). Missing keys degrade gracefully to English.
3. Use `t('your.new.key')` at every call-site.

The `TranslationKey` type in `config.ts` is derived from `en.json`, so typos at call-sites become build-time errors.

## 7. RTL (right-to-left)

The launch locales (`en`, `hi`, `ta`, `bn`) are all LTR. However, we test the CSS under a forced-RTL context (Chrome DevTools → Rendering → "Emulate CSS media type: rtl") to confirm we don't break layout if a user's browser is set to Arabic or Hebrew:

- Every component uses logical properties (`margin-inline-start` rather than `margin-left`) where feasible.
- Icons that carry a directional cue (chevrons, arrows) mirror in RTL.
- The primary layout container has `dir` set from `document.documentElement.lang`.

Full Arabic / Hebrew coverage is a Phase H+ task — the scaffold merely does not break.

## 8. Testing

- **Unit** — `translate()` fallback logic is a pure function. Add tests when we onboard the next locale.
- **Visual** — snapshot Discover / Match modal / Settings under each locale to catch clipped labels (Hindi in particular is 20–30% longer than English).
- **Manual** — the Settings language switcher is the fastest way to eyeball any surface in Hindi.

## 9. What is NOT in scope for this launch

- Server-side locale detection (Accept-Language header) — the client-side switcher plus localStorage is enough for launch.
- ICU pluralisation rules — deferred to Phase H+ once we have a real translator in loop.
- Locale-aware URL routing (`/hi/discover`) — deferred; the app is single-locale-per-user at the state level, not per URL.
- Machine-translated fallbacks — the fallback is always to English until a human translation ships.
- Full-coverage translation of every existing component — the scaffold plus a top-30-string set is what ships today. Systematic conversion of the rest of the app is a Phase H+ task tracked separately.
